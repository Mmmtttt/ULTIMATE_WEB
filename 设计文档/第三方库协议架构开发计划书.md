# 第三方库协议架构开发计划书

更新时间：2026-04-19  
适用范围：`ULTIMATE_WEB` 第三方库协议化重构实施方案

---

## 1. 项目背景

当前项目的第三方库接入已经覆盖漫画、视频、在线播放、收藏夹同步、远程导入、`LOCAL` 元数据补全等大量功能，但接入方式仍然处于“历史演进叠加”的状态：

1. 有统一适配器体系。
2. 也有直接 import 具体第三方实现的路径。
3. 有部分 schema 驱动配置。
4. 也有平台名写死的宿主逻辑。

这导致新增第三方库或新增能力时，往往需要同时修改：

1. 第三方库代码。
2. 宿主侧平台注册逻辑。
3. 宿主侧配置接口。
4. 前端配置界面。
5. 业务层平台判断。

本次重构的目标，是把这件事改造成“协议驱动”模式：

1. 宿主只认协议，不认具体第三方库。
2. 第三方库自己声明能力，自己实现统一 provider。
3. 宿主以能力路由的方式完成调用。
4. 在保证当前功能完全不变的前提下完成迁移。

---

## 2. 项目目标

### 2.1 核心目标

1. 保持现有用户功能、前端页面、后端 API 行为完全不变。
2. 建立统一的第三方插件协议。
3. 建立统一的宿主协议内核。
4. 把主要适配层下放到第三方库内部。
5. 支持未来方便地增减第三方库和功能。

### 2.2 验收标准

满足以下条件视为重构完成：

1. 宿主新增一个第三方库时，不需要再修改宿主业务代码。
2. 宿主的业务代码不再出现平台名硬编码分支。
3. 配置页面可根据插件清单自动生成。
4. 现有第三方相关测试全部通过。
5. 漫画、视频、在线播放、收藏夹、导入、元数据补全等现有链路行为保持不变。

### 2.3 非目标

第一阶段暂不追求：

1. 把第三方插件做成独立进程沙箱。
2. 修改前端对外接口形态。
3. 重写第三方库原始爬虫/播放器核心逻辑。
4. 立刻清理所有历史兼容代码。

## 4. 目标架构

### 4.1 宿主侧模块

本次重构建议在宿主新增或重构以下模块：

1. `plugin_registry`
   扫描插件目录，读取 `manifest`，建立插件索引。
2. `provider_manager`
   负责 provider 加载、配置注入、热重载。
3. `capability_router`
   负责能力到插件的路由和执行策略。
4. `protocol_gateway`
   作为业务层统一调用入口。
5. `plugin_config_service`
   负责配置保存、脱敏、schema 输出。
6. `compatibility_bridge`
   兼容当前老接口，保证迁移期稳定。

### 4.2 第三方库侧模块

每个第三方库建议增加：

1. `ultimate-plugin.json`
2. `ultimate_provider.py`
3. `translator.py`
4. `schemas/`

### 4.3 分层边界

必须明确划清责任：

宿主负责：

1. 本地数据库。
2. API 输出。
3. 任务系统。
4. 缓存与数据目录策略。
5. UI 配置渲染。

插件负责：

1. 上游数据获取。
2. 认证方式。
3. 上游错误翻译。
4. 结果标准化。
5. 能力声明。

## 5. 实施总原则

### 5.1 双轨迁移

整个项目必须采用“双轨迁移”：

1. 先建新协议内核。
2. 再让老业务逐步切换到新协议。
3. 在迁移完成前，保留兼容桥接层。

这样可以最大限度降低对现有功能的破坏。

### 5.2 测试先行

不能先大改再补测试。  
当前已有的第三方集成测试必须先作为保护网固定下来，再做迁移。

### 5.3 前端接口不动

第一阶段不改前端 API 契约。  
这样可以把风险集中在后端内部结构，不把前后端一起拖进重构。

## 6. 分阶段开发计划

## 阶段 0：现状冻结与契约基线

### 目标

把当前所有第三方行为冻结成“现有契约”，作为后续重构的回归标准。

### 工作内容

1. 梳理当前第三方相关 API 和业务入口。
2. 补齐第三方能力矩阵文档。
3. 盘点现有集成测试和 E2E 测试。
4. 新增缺失的协议级回归测试。

### 产出

1. 能力矩阵。
2. 契约测试清单。
3. 回归基线报告。

### 验收

1. 现有第三方集成测试全部可稳定运行。
2. 每条核心功能都有对应测试覆盖。

## 阶段 1：协议模型与宿主内核落地

### 目标

先把“协议框架”搭起来，但暂时不切业务。

### 工作内容

1. 新建协议模型：
   - `manifest`
   - `capability`
   - `request/response`
   - `error`
   - `configuration`
2. 实现 `PluginRegistry`。
3. 实现 `ProviderManager`。
4. 实现 `CapabilityRouter`。
5. 实现 `ProtocolGateway`。
6. 实现插件配置存储与 schema 输出。
7. 实现宿主路由配置文件。

### 推荐代码落点

建议新增目录：

```text
comic_backend/
  protocol/
    models.py
    manifest_loader.py
    provider_manager.py
    capability_router.py
    gateway.py
    config_service.py
    compatibility_bridge.py
```

### 产出

1. 协议内核可独立运行。
2. 宿主可以扫描并加载插件。
3. 宿主可以执行统一 `gateway.execute()`。

### 验收

1. 使用一个最小 demo 插件即可跑通加载与调用。
2. 配置页面可以从协议 schema 读取并渲染。

## 阶段 2：把现有第三方库包装成协议插件

### 目标

不改业务逻辑，先让现有第三方库都具备协议插件形态。

### 工作内容

1. 为 `JMComic-Crawler-Python` 增加：
   - `ultimate-plugin.json`
   - `ultimate_provider.py`
2. 为 `Picacomic-Crawler` 增加相同结构。
3. 为 `javdb-api-scraper` 增加相同结构。
4. 为 `Missav` 增加相同结构。
5. 如需保留 `javbus`，则单独做成一个插件或作为 `javdb-api-scraper` 下的多能力 provider。

### 实施要求

1. 插件内部完成原始返回值到标准模型的转换。
2. 插件内部完成原始异常到协议错误码的映射。
3. 插件内部完成配置格式的兼容转换。

### 产出

1. 每个现有第三方库都能通过协议网关独立调用。
2. 每个现有第三方库都能输出自己的配置 schema。

### 验收

1. 不通过老的 `platform_service` 也能直接调用成功。
2. 协议插件对同一功能输出与旧路径一致。

---

## 阶段 3：漫画链路切换到协议网关

### 目标

先迁移漫画，因为当前漫画路径相对集中，可作为第一批验证对象。

### 工作内容

1. 漫画第三方搜索改走 `ProtocolGateway`。
2. 漫画详情获取改走 `ProtocolGateway`。
3. 收藏夹与清单同步改走 `ProtocolGateway`。
4. 下载封面、下载整本漫画、预览图获取改走 `ProtocolGateway`。
5. `LOCAL` 漫画补全改走 `ProtocolGateway`。
6. 作者远程作品获取链路改走 `ProtocolGateway`。

### 迁移策略

先保留旧函数名，对内改为委托给新协议网关。

例如：

1. `platform_service.search_albums()` 内部先改为调 `gateway.execute(...)`
2. `external_api.search_albums()` 内部也改为调 `gateway.execute(...)`

这样业务层基本不用一次性全改。

### 验收

1. 漫画第三方相关测试全部通过。
2. 配置页面中的漫画平台配置仍正常工作。

## 阶段 4：视频元数据链路切换到协议网关

### 目标

迁移视频搜索、详情、演员、标签、清单、推荐流等非播放能力。

### 工作内容

1. 把 `api/v1/video.py` 中直接平台实例化逻辑替换为协议网关。
2. 把 `get_video_adapter()` 改造成兼容桥。
3. 演员、标签、清单、导入等链路都改成能力调用。
4. 预览图、封面、预览视频抓取改为资源能力调用。

### 风险点

视频链路当前平台分支和能力差异最多，必须重点回归：

1. 多平台搜索合并结果。
2. 标签搜索分页。
3. 推荐流导入。
4. 清单与收藏夹同步。

### 验收

1. 视频第三方集成测试全部通过。
2. 视频标签搜索与演员链路行为不变。

## 阶段 5：在线播放链路切换到协议网关

### 目标

把 `Missav` 相关播放能力也纳入统一协议。

### 工作内容

1. 把 `build_sources` 改成 `playback.sources.build`。
2. 把代理转发改成 `playback.proxy.forward`。
3. 统一播放错误返回格式。
4. 统一播放链路的认证与配置读取。

### 风险点

播放链路是最容易引入隐性回归的部分，必须重点验证：

1. m3u8 代理。
2. key 代理。
3. ts 分片代理。
4. referer/header/cookie 透传。

### 验收

1. 播放代理契约测试全部通过。
2. 现有在线播放体验保持一致。

## 阶段 6：配置中心与前端彻底协议化

### 目标

把当前写死平台名的配置结构，升级成真正的插件协议配置中心。

### 工作内容

1. 把当前 `comic.py` 中写死的第三方 schema 迁移为从插件清单读取。
2. 把前端 `SystemConfig.vue` 的平台列表来源改为协议插件列表。
3. 增加配置交互动作渲染：
   - 打开教程页
   - 校验认证
   - 显示认证状态
4. 支持 secret 脱敏回显。
5. 支持启用/禁用、排序与路由优先级配置。

### 验收

1. 新增一个插件后，配置页面自动出现对应配置项。
2. 不再需要手工改后端 schema 构造逻辑。

## 阶段 7：收口与清理

### 目标

在所有链路切换完成后，清理历史遗留接入方式。

### 工作内容

1. 清理宿主业务代码中的平台硬编码分支。
2. 清理不再需要的旧适配入口。
3. 保留必要的兼容桥，但标记为 deprecated。
4. 输出最终维护文档和接入手册。

### 清理对象

重点包括：

1. 重复的 `platform_service` 实现。
2. 业务层对具体第三方类的直接 import。
3. 写死平台名的配置 schema。

### 验收

1. 宿主业务层不再直接 import 第三方平台实现。
2. 新功能接入只需要新增插件，不需要改宿主业务代码。

## 7. 关键技术决策

### 7.1 不采用“纯 JSON 执行协议”

原因：

1. JSON 只能声明，不能执行业务逻辑。
2. 只用 JSON 会迫使宿主再次实现平台差异分发。
3. 这和“适配层下放到第三方库内部”的目标相冲突。

所以必须采用：

1. `manifest JSON` 负责声明。
2. `provider` 负责执行。

### 7.2 业务层按“能力”组织，而不是按“平台”组织

后续业务层应从：

1. “去调用 `javdb`”

改为：

1. “去调用 `video.catalog.detail` 能力”

这是整个架构能否真正解耦的核心。

### 7.3 迁移期保留兼容桥

不能一次性强切，否则风险过大。  
兼容桥可以让旧接口继续存在，但内部委托给新协议网关。

## 8. 风险与应对

### 风险 1：功能细节回归

原因：

1. 第三方能力非常碎。
2. 某些平台有隐式行为，例如轻量收藏夹、标签分页、Cookie 补全、播放 header 透传。

应对：

1. 先补齐契约测试。
2. 分链路迁移。
3. 漫画先迁，视频后迁，播放最后迁。

### 风险 2：标准模型过度抽象，丢失细节

原因：

1. 视频和漫画差异大。
2. 同一媒体内各平台返回结构也不一致。

应对：

1. 标准模型只抽“共性必需字段”。
2. 插件特有字段放进 `extra`。

### 风险 3：配置协议过重

原因：

1. 如果第一版追求过于动态，会大幅提高实现成本。

应对：

1. 第一版只支持固定字段类型和固定动作类型。
2. 后续再扩展。

### 风险 4：一次性改动面过大

应对：

1. 必须保留兼容桥。
2. 必须用 feature flag 控制切换。

建议新增运行开关：

1. `THIRD_PARTY_PROTOCOL_V2_ENABLED=false`

默认先关闭，按模块灰度开启。

## 10. 建议实施顺序

如果按投入产出比排序，建议实际开发顺序如下：

1. 先做协议内核和配置中心。
2. 再做 JM、PK 两个漫画插件。
3. 然后迁移漫画所有调用链。
4. 再做 JAVDB/JAVBUS 视频元数据插件。
5. 然后迁移视频搜索、详情、演员、标签、清单链路。
6. 最后纳入 Missav 播放协议。
7. 全量稳定后再做历史代码清理。

这样最稳，也最符合“保证当前功能完全不变”的要求。

## 12. 最终结论

这次重构不是单纯“把 adapter 改个名字”，而是要把第三方接入从“代码硬编码平台”升级为“协议驱动能力”。

最合理的实施方式是：

1. 先定义统一协议。
2. 再实现宿主协议内核。
3. 然后把现有第三方库逐个包装成协议插件。
4. 通过兼容桥逐步替换现有业务调用链。
5. 用现有第三方集成测试做全程护栏。

只要严格按这个顺序推进，就可以在不改变当前用户功能的前提下，把第三方库接入彻底重构成可扩展、可维护、可持续演进的架构。
