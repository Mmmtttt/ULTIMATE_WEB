# 多架构兼容项目改造可行性分析与架构设计说明书

## 1. 项目背景与架构改造目的

本系统现两种部署架构：
- 架构1：Windows环境部署，支持内网IP访问
- 架构2：Linux环境部署，支持公网IP访问

本次架构改造旨在新增第三种部署形态并实现多架构统一开发与构建：
- 架构3：前后端打包为Android APK应用，运行于安卓环境
- 实现跨架构统一代码库，确保后续开发对架构差异无感知
- 构建系统支持一次打包输出三种部署包：Android APK、Windows可执行程序(.exe)、Linux可执行程序

## 2. 架构设计原理

### 2.1 核心设计原则
- **架构透明化**：业务逻辑代码与架构适配代码分离，确保业务开发对底层架构无感知
- **模块化隔离**：将平台相关功能与通用功能解耦，通过接口抽象实现平台适配
- **数据同步机制**：架构3环境下apk拥有除第三方库之外的完整后端能力，可以通过网络连接架构1/2的后端服务器同步数据
- **构建自动化**：建立统一构建流程，支持一次构建输出多平台部署包

### 2.2 多架构适配策略
- **前端适配**：采用响应式设计，确保UI在不同设备与环境下正常展示
- **后端适配**：
  - 架构1/2保持完整功能（含第三方库支持）
  - 架构3剔除第三方库依赖，仅保留基础业务逻辑与数据同步功能
- **数据同步**：架构3通过网络API连接架构1/2后端，实现数据双向同步

## 3. 代码设计与模块变更分析

### 3.1 前期代码检查要点
1. **前端代码检查**：
   - 框架兼容性：检查是否使用浏览器特定API，评估迁移至Android WebView环境的可行性
   - 本地存储方案：分析localStorage/sessionStorage使用情况，评估在Android环境下的替代方案
   - 网络请求：检查API调用方式，评估在不同架构环境下的URL配置策略

2. **后端代码检查**：
   - 第三方库依赖分析：识别无法迁移至Android的第三方库，评估隔离方案
   - 平台特定代码：检查是否包含Windows/Linux平台特定代码，评估抽象适配方案
   - 数据访问层：评估数据访问逻辑与业务逻辑的耦合程度，规划数据同步接口

### 3.2 架构改造关键模块设计
1. **架构适配层**：
   - 设计平台检测机制，实现不同架构环境的自动识别
   - 抽象网络请求接口，根据当前架构自动切换请求目标（本地后端/远程服务器）

2. **数据同步模块**：
   - 设计数据同步协议，支持增量同步与全量同步
   - 实现离线数据缓存机制，确保架构3在无网络环境下的基础功能可用

3. **构建系统**：
   - 配置多平台构建脚本，支持条件编译与资源打包
   - 实现架构特定代码的自动引入与排除

## 4. 注意事项

### 4.1 技术可行性注意事项
- **第三方库迁移限制**：需彻底隔离无法迁移的第三方库，确保其仅在架构1/2中被引用
- **Android环境限制**：评估WebView对前端技术的支持程度，确认无兼容性问题
- **性能考量**：架构3环境下需优化网络请求策略，减少数据同步对移动端性能的影响

### 4.2 架构变更后开发注意事项
- **代码规范**：
  - 新增平台相关代码必须通过架构适配层封装，禁止直接在业务逻辑中使用平台特定API
  - 所有网络请求必须使用统一封装的请求接口，禁止硬编码URL

- **测试要求**：
  - 新功能必须在三种架构环境下进行测试验证
  - 建立自动化测试流程，确保跨架构兼容性

- **构建流程**：
  - 严格遵循统一构建流程，确保一次构建输出所有平台部署包
  - 版本号管理需保持各平台部署包版本一致性

- **数据同步**：
  - 设计合理的数据同步策略，平衡数据实时性与网络消耗
  - 实现数据冲突解决机制，处理多端数据同步冲突

## 5. 实施步骤建议
1. 完成现有代码的全面评估，确认可迁移性与需修改的模块
2. 设计并实现架构适配层与数据同步模块

4. 改造现有代码以适配新架构设计

# 多架构兼容项目改造可行性分析与架构设计说明书

> 适用项目：ULTIMATE_WEB（Vue3 + Flask）
> 
> 目标：在现有 Windows / Linux 部署基础上，新增 Android APK 形态，并建立“一套代码、三端构建”的统一架构。
>
> 评估基线：commit `774ab6d`。

---

## 1. 项目背景与架构改造目的

### 1.1 当前代码现状（基于仓库代码评估）

项目已具备前后端分离架构：

- 后端为 Flask，入口为 `comic_backend/app.py`，通过 `server_config.json` 读取运行地址与端口，并已包含 `storage.data_dir`；默认绑定 `0.0.0.0`，具备局域网访问能力。  
- 前端为 Vue3 + Vite，开发环境通过 `vite.config.js` 使用 `/api` 与 `/static` 代理到后端。  
- 工程同时提供 Windows (`*.ps1`) 与 Linux (`*.sh`) 启停脚本，说明“架构1/架构2”在工程层面已被显式支持。

### 1.2 改造目标

在不破坏现有功能的前提下，新增：

1. **架构3（Android APK）**：前端运行于 Android WebView / Hybrid 容器；后端能力通过网络连接架构1/2服务器获取。  
2. **统一代码库**：业务层不直接感知平台差异，平台分歧收敛到适配层。  
3. **统一构建**：一次流水线可输出 Android APK、Windows 可执行包、Linux 可执行包。

### 1.3 可行性总体结论

**总体可行（中高可行），但需分阶段改造。**

- 前端已具备移动端友好基础（Vant + 响应式能力），适配 Android WebView 成本可控。  
- 后端已存在分层（api/application/domain/infrastructure）与适配器模式雏形（`third_party/*`），有利于平台隔离。  
- 但目前仍存在多处“平台直连点”（例如直接使用 `localStorage`、`window/document`、双请求封装并存），需要先完成“架构治理”再引入 APK 打包。
  - 代码例子A（直接 `localStorage`）：`comic_frontend/src/stores/mode.js` 中直接读写 `localStorage.getItem/setItem`。
  - 代码例子B（直接 `window/document`）：`comic_frontend/src/views/ComicReader.vue`、`comic_frontend/src/views/RecommendationReader.vue` 中直接使用 `window.innerWidth`、`document.addEventListener`、`requestFullscreen`。
  - 代码例子C（双请求封装并存）：`comic_frontend/src/api/request.js` 与 `comic_frontend/src/utils/request.js` 同时存在 `axios.create(...)`，且 baseURL 计算策略不同。
- 后端数据目录改造在该基线节点已完成关键落地：`server_config.json` 提供 `storage.data_dir`，`core/constants.py` 由配置解析 `DATA_DIR` 并派生 `META_DIR/PICTURES_DIR/STATIC_DIR/COVER_DIR`，第三方下载路径支持归一化。
  - 代码例子A（配置入口）：`server_config.json` 包含 `storage.data_dir`。
  - 代码例子B（目录解析）：`comic_backend/core/constants.py` 通过 `_resolve_data_dir()` 计算 `DATA_DIR`。
  - 代码例子C（路径归一化）：`comic_backend/core/constants.py` 提供 `normalize_to_data_dir(...)`。
  - 代码例子D（业务使用常量）：`comic_backend/infrastructure/task_manager.py` 默认任务文件改为 `IMPORT_TASKS_JSON_FILE`，`comic_backend/app.py` 封面路由使用 `COVER_DIR`。
- 视频代理与抓流能力在该基线节点已完成主要收敛：`api/v1/video.py` 通过 third-party 客户端代理，`curl_cffi` 主要位于 third-party 实现。

## 2. 架构设计原理

### 2.1 核心设计原则

1. **架构透明化**
   - 业务代码只依赖统一接口：`IPlatformRuntime`、`IApiGateway`、`IDataSyncService`。
   - 平台差异（Windows/Linux/Android）由 adapter 实现，业务层禁止直接读取 `window`、`localStorage`、硬编码 URL。

2. **模块化隔离**
   - 将后端“可跨平台核心能力”与“不可迁移第三方能力”分层：
     - Core：元数据管理、标签清单、阅读进度、缓存与同步协议；
     - Plugin：JM/JAVDB 等第三方采集器与视频代理能力，仅在架构1/2启用；架构3默认关闭。

3. **数据同步机制**
   - 架构3不承载完整采集后端，改为“轻客户端 + 同步中台”模式。
   - 使用增量游标（`updated_at` / `version`）进行双向同步，支持离线缓存与冲突解决。

4. **构建自动化**
   - 前端统一产物（Web 静态包）复用；
   - Windows/Linux 通过 Python 打包工具（PyInstaller/Nuitka）出可执行；
   - Android 通过 Capacitor/Tauri Mobile（推荐 Capacitor）生成 APK。

### 2.2 多架构适配策略

#### 前端适配

- 现有响应式机制（`useDevice`）可继续保留，但需补充“运行时平台识别”与“能力探测”。
- 将浏览器 API 访问统一封装到 `runtime` 层：
  - `ClipboardService`
  - `ShareService`
  - `FullscreenService`
  - `StorageService`

#### 后端适配

- 架构1/2：保留全部现有功能（包括 `third_party` 采集与封面抓取）。
- 架构3：
  - 不运行采集链路；
  - 仅保留 API 客户端、缓存、同步、基础阅读与配置能力。

#### 数据同步

- 架构3通过 HTTPS API 连接架构1/2后端：
  - 上行：阅读进度、收藏/标签改动、用户配置；
  - 下行：媒体元数据、封面、推荐列表、配置。
- 同步模式：
  - 前台轻量增量同步（启动/前后台切换）；
  - 后台定时同步（网络可用且电量允许时）；
  - 异常时回退离线缓存。

## 3. 代码设计与模块变更分析

## 3.1 前期代码检查结论

### 3.1.1 前端检查结论

1. **框架兼容性**：
   - 使用 Vue3/Vite/Vant，无明显阻塞 Android WebView 的框架依赖；
   - 但存在大量 `window/document/navigator` 直接调用（阅读器全屏、拖拽、剪贴板、分享），需通过适配层兜底。

2. **本地存储方案**：
   - 同时存在封装存储（`src/utils/storage.js`）与直接 `localStorage/sessionStorage` 访问（如 `stores/mode.js`、`views/ListDetail.vue`、`utils/constants.js`），规范未统一；
   - Android WebView 可支持 Web Storage，但建议统一迁移至 `StorageService`，为后续切换 SQLite/Preferences 预留接口。

3. **网络请求**：
   - 当前存在 `src/api/request.js` 与 `src/utils/request.js` 两套 axios 封装，且 baseURL策略不一致；
   - Android 场景需统一“环境路由”（本地代理 / 远程网关 / HTTPS证书策略），建议收敛为单一请求网关。

### 3.1.2 后端检查结论

1. **第三方依赖迁移限制**：
   - 后端依赖中包含 `jmcomic`、`Pillow`、`requests`，并在 `third_party` 与 `application` 中深度使用；
   - 基线节点中视频代理已通过 third-party 客户端调用，API 层主要负责路由编排；架构3仍应通过 feature flag 默认关闭该能力。

2. **平台特定代码**：
   - 启停脚本已分 Windows/Linux；
   - 当前主要风险集中在两类：前端运行时 API 直连、请求网关双入口并存。

3. **数据访问层耦合**：
   - 已有 domain/application/infrastructure 分层与仓储实现，具备继续抽象条件；
   - 但当前数据持久化仍以 JSON 文件为主，跨设备同步需额外“变更日志/版本号”机制支撑。

### 3.2 架构改造关键模块设计

#### 3.2.1 架构适配层（新增）

建议新增目录（前端）：

```text
comic_frontend/src/platform/
  runtime/
    index.js                # 检测平台(web/android/desktop)
    capability.js           # 能力探测（clipboard/share/fullscreen/storage）
  adapters/
    web.adapter.js
    android.adapter.js
  gateway/
    request-gateway.js      # 全局唯一请求入口
    endpoint-resolver.js    # 根据平台/配置解析 API 地址
```

建议新增目录（后端）：

```text
comic_backend/architecture/
  feature_flags.py          # 运行模式：full / sync-only
  dependency_profile.py     # third_party 可用性与禁用策略
```

关键机制：

- 通过 `APP_RUNTIME_TARGET`（web/windows/linux/android）与 `BACKEND_MODE`（full/sync-only）驱动条件行为；
- 业务层只注入接口，不直接判断平台字符串。

#### 3.2.2 数据同步模块（新增）

建议新增协议对象（并将路径输入改为配置解析后的目录角色，而非硬编码目录）：

- `sync_cursor`: 上次同步时间戳 + 逻辑版本；
- `sync_ops`: 本地待上行操作队列（append-only）；
- `conflict_policy`: `server_wins`（默认）+ 针对阅读进度的 `max(page,last_read_time)`。
- `storage_roots`: 同步服务端当前使用的目录角色映射（`meta_root`/`pictures_root`/`cover_root`/`cache_root`）。

建议新增 API：

- `GET /api/v1/sync/pull?cursor=xxx`
- `POST /api/v1/sync/push`
- `POST /api/v1/sync/resolve`

离线策略：

- 本地写入先落缓存 + 入同步队列；
- 网络恢复后重放队列并更新 cursor；
- 失败重试采用指数退避。

#### 3.2.3 构建系统（重构）

建议建立统一构建入口：

```text
build/
  build_all.sh / build_all.ps1
  targets/
    build_web.sh
    build_windows.sh
    build_linux.sh
    build_android.sh
```

构建建议：

1. 前端：`npm run build` 输出 `dist`。  
2. Windows/Linux：
   - 后端用 PyInstaller 打包；
   - 产物目录中嵌入 `dist` 静态资源（可选 Nginx/内置服务）。
3. Android：
   - 采用 Capacitor 打包前端 `dist`；
   - APP 启动后通过配置中心或首次引导设置服务器地址。


#### 3.2.4 当前迁移阻塞点、难点与改动清单（按优先级）

**P0 阻塞点（不解决就不建议推进 Android 架构接入）**

1. **前端请求网关双入口并存**
   - 现状：`src/api/request.js` 与 `src/utils/request.js` 同时创建 axios 实例，baseURL 策略不同。
   - 影响：Android/公网切换时请求目标不一致，容易出现“部分接口可用、部分不可用”。
   - 必改：保留一个网关入口，所有 API 统一走 endpoint-resolver。

2. **浏览器能力直连未封装**
   - 现状：多处直接 `window/document/localStorage/navigator`。
   - 影响：WebView 能力差异将直接外溢到业务组件。
   - 必改：引入 runtime/capability 层并替换业务直连调用。

3. **同步协议尚未落地**
   - 现状：文档定义了 `sync/session/manifest/download`，代码中尚无对应 API。
   - 影响：Android 端“手动同步 + 断点恢复”无法真正上线。
   - 必改：先实现最小 API 与 manifest 任务状态机，再接入客户端。

**P1 难点（可并行推进）**

4. **同步导入任务可靠性（中断/恢复/幂等）**
   - 现状：尚未看到同步任务状态机与导入幂等实现。
   - 影响：弱网、闪退、断电场景下容易产生半导入状态。
   - 必改：引入 `manifest.import_state.json`、包校验、阶段提交与重放机制。

5. **Android 端服务器配置与环境切换**
   - 现状：当前方案是手动输入服务器地址，但缺少统一配置校验流程与错误分级。
   - 影响：公网/内网切换时易出现排障困难。
   - 必改：固化 `health + version` 探测、配置持久化与可观测日志。

**P2 工程化项（不阻塞设计，但阻塞量产）**

6. **多平台一键构建脚本缺失**
   - 现状：文档有 `build_all` 规划，仓库暂无对应脚本与 Android 工程产线。
   - 影响：无法做到一次流水线输出三端制品。
   - 必改：补齐 build 目录、CI 任务、版本注入规则。

## 4. 注意事项

### 4.1 技术可行性注意事项

1. **第三方库迁移限制**
   - `third_party` 能力必须拆分为可选插件，并在 android/sync-only 模式下禁用。

2. **Android WebView 限制**
   - `requestFullscreen`、`navigator.share`、`clipboard` 在不同 WebView 版本兼容性不一致，必须提供降级路径。

3. **性能与流量**
   - 图片和视频元数据同步要分层（元数据先行、媒体按需）；
   - 需增加请求压缩、etag/cache-control、批量接口。

### 4.2 架构变更后开发注意事项

1. **代码规范**
   - 禁止在业务组件内直接调用 `window/document/localStorage/navigator`；
   - 禁止出现第二个 axios 实例入口；
   - URL 一律通过 `endpoint-resolver` 解析；
   - 同步/备份/导入导出代码禁止硬编码 `data/...`、`static/...`，统一从路径配置解析层获取目录。

2. **测试要求**
   - 三架构最小回归矩阵：
     - 启动/登录（如有）
     - 列表加载
     - 详情打开
     - 阅读进度保存
     - 标签/清单修改
     - 同步冲突处理

3. **构建流程**
   - CI 单 pipeline 产出三类制品；
   - 三端版本号同源（例如 `YYYY.MM.DD+build`），构建时自动注入。

4. **数据同步**
   - 冲突优先级先做简单策略（服务器优先 + 阅读进度取最大），后续迭代 CRDT/字段级合并。

## 5. 实施步骤建议（分阶段）

### 阶段A：可迁移性治理（1~2 周）

- 收敛前端请求网关为唯一入口并统一 baseURL 解析。
- 抽离并替换浏览器能力直连（window/document/localStorage/navigator）。
- 后端引入 `BACKEND_MODE` 并把 third_party 能力默认按架构裁剪。

### 阶段B：同步基础设施（2~3 周）

### 阶段C：多端构建流水线（1~2 周）

- Web + Windows + Linux 统一构建脚本（补齐 `build_all` 与 targets 脚本）；

### 阶段D：联调与验收（1~2 周）

- 三端回归测试与兼容性修复；

- 发布规范与运维文档定稿。

## 6. 风险清单与应对

1. **风险：第三方采集链路与核心业务耦合**  
   应对：以 feature flag 和接口隔离，先“禁用可运行”再“逐步解耦”。

2. **风险：Android 端能力不一致**  
   应对：能力探测 + 降级实现，关键功能提供原生桥接插件。

3. **风险：JSON 存储难支撑多端一致性**  
   应对：短期增加版本字段与变更日志；中期评估 SQLite/PostgreSQL 服务化。

4. **风险：构建链过长导致发布复杂**  
   应对：统一版本、统一命名、统一制品仓库，建立发布检查清单。

## 7. 结论

- 本项目具备多架构改造基础，**建议按“治理先行、同步其次、构建最后”路线实施**。  
- 在不大改业务逻辑的情况下，可实现 Android 新架构接入与三端统一构建。  
- 核心成功因素在于：**适配层强约束 + 请求网关统一 + 同步协议稳健 + CI/CD 一次出包**。


## 8. 数据同步模块详细实现方案（按当前决策落地）

> 本章节基于当前已确认策略：
> 1) 纯手动填写服务器地址；
> 2) 暂不引入全面安全鉴权；
> 3) 图片按漫画目录级做增量；
> 4) 所有包下载完成后再导入，先大包后 meta；
> 5) 用户手动点击“同步”触发。

### 8.1 目标与边界

- **目标**：以最小复杂度实现“可恢复、可中断、可重试”的手机端同步。
- **边界**：本阶段不做自动后台同步，不做账号体系，不做复杂权限模型。
- **原则**：宁可多一次全量下载，也不冒数据一致性风险。

### 8.2 同步对象分层与打包策略

服务端将同步数据拆为四层，并生成会话级快照（snapshot）：

1. `media`（大包/多包，目录级增量）
   - 来源：`PICTURES_ROOT/*`（由后端配置解析得出）
   - 粒度：按“漫画目录”拆包（例如 `media_chunk_0001.zip`）。
   - 规则：只打“手机端缺失且服务端存在”的目录。

2. `cache`（可选中包）
   - 来源：`RECOMMENDATION_CACHE_ROOT/*`、`CACHE_ROOT/*`（由后端配置解析得出）
   - 规则：可按开关选择是否下发（弱网可关闭）。

3. `cover`（小中包）
   - 来源：`COVER_ROOT/*`（由后端配置解析得出）
   - 规则：默认全量覆盖（体积可控）。

4. `meta`（小包，最后导入）
   - 来源：`META_ROOT/*`（由后端配置解析得出）
   - 规则：最后导入，确保索引与前面已导入资源一致。

### 8.3 连接与交互设计（手动服务器地址）

#### 8.3.1 手机端服务器配置

新增“同步服务器配置”页面：

- 字段：`server_url`（必填），`nickname`（可选），`timeout`（可选）。
- 示例：
  - 局域网：`http://192.168.1.20:5000`
  - 公网：`https://example.com`
- 校验流程：
  1) `GET {server_url}/health`
  2) `GET {server_url}/api/v1/sync/version`（检查协议版本）
  3) 通过后写入本地 `sync_config.json`

#### 8.3.2 交互流程（手动触发）

用户点击“立即同步”后：

1. 创建会话：`POST /api/v1/sync/session/start`
2. 拉取清单：`GET /api/v1/sync/manifest?session_id=...`
3. 全量下载：按 manifest 下载所有 zip（支持断点）
4. 全部校验：sha256 全通过后进入导入
5. 分阶段导入：`media -> cache -> cover -> meta`
6. 提交完成：`POST /api/v1/sync/session/finish`

### 8.4 manifest 与任务状态机

#### 8.4.1 manifest 结构（核心）

```json
{
  "schema_version": 1,
  "session_id": "20260316_abcdef",
  "snapshot_version": "2026-03-16T10:10:10Z",
  "snapshot_generated_at": "2026-03-16T10:10:12Z",
  "server_data_root_fingerprint": "sha256:xxxx",
  "packages": [
    {
      "id": "media_chunk_0001",
      "type": "media",
      "file": "media_chunk_0001.zip",
      "size": 123456789,
      "sha256": "...",
      "priority": 10
    },
    {
      "id": "meta",
      "type": "meta",
      "file": "meta.zip",
      "size": 12345,
      "sha256": "...",
      "priority": 100
    }
  ]
}
```

#### 8.4.2 手机端任务状态

- `CREATED`：会话已创建
- `DOWNLOADING`：下载中
- `DOWNLOADED`：全部包下载完成
- `VERIFYING`：校验中
- `IMPORTING_MEDIA/CACHE/COVER/META`：分阶段导入
- `COMPLETED`：完成
- `FAILED_RETRYABLE`：可重试失败
- `FAILED_FATAL`：致命失败（需人工处理）

### 8.5 导入顺序与中断恢复（按你确认方案）

你提出“先大包后 meta”是正确的，能显著降低中断影响。建议细化为：

1. **下载阶段**：必须“全部包下载完成 + 全量 hash 校验通过”后才允许导入。
2. **导入阶段（固定顺序）**：
   - `media`（最大体积，最容易中断，先处理）
   - `cache`
   - `cover`
   - `meta`（最后一步，索引收口）
3. **阶段记录**：每完成一个包，就把结果写入本地 `manifest.import_state.json`。
4. **重启恢复**：启动时读取 `manifest.import_state.json`，从未完成包继续。
5. **最终提交**：仅当 `meta` 导入完成后，将 `state=COMPLETED`。

> 这样即使中断，也会停留在“旧 meta + 新资源”状态，不会出现“新 meta 指向不存在资源”的严重不一致。

### 8.6 服务端最小 API 设计（V1）

1. `POST /api/v1/sync/session/start`
   - 入参：手机端已有目录摘要（可选）
   - 出参：`session_id`、`snapshot_version`

2. `GET /api/v1/sync/manifest?session_id=...`
   - 出参：包清单、下载地址、hash、顺序

3. `GET /api/v1/sync/download/<package_id>`
   - 要求：支持 `Range`，便于断点续传

4. `POST /api/v1/sync/session/finish`
   - 入参：完成状态、失败包列表（如有）

### 8.7 异常场景与处理

1. **网络中断**：下载任务自动重试（指数退避），保留已下载分片。
2. **包校验失败**：删除损坏包并重下。
3. **存储空间不足**：下载前做容量预检；不足则阻止同步并提示清理空间。
4. **导入中断（闪退/断电）**：根据 `manifest.import_state.json` 续导。
5. **服务端会话过期**：提示“会话失效，请重新发起同步”。
6. **版本不兼容**：`schema_version` 不一致时拒绝导入。

### 8.8 与现有代码衔接建议

- 后端可在现有 `api/v1` 下新增 `sync.py` 蓝图，保持与现有 `backup.py` 风格一致。
- 数据目录分层需先完成“配置解析层统一输出目录角色”后再接入同步打包；在此之前，同步模块不得新增硬编码目录。
