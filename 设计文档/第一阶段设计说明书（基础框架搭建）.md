## 2. 模块功能设计

### 2.1 路由层（Routes）

#### 2.1.1 模块职责
- 接收HTTP请求，解析请求参数
- 调用对应工具类完成业务逻辑
- 封装统一响应格式
- 异常捕获与处理

#### 2.1.2 接口清单
| 接口 | 方法 | 路径 | 功能说明 |
|------|------|------|----------|
| 漫画初始化 | POST | /api/v1/comic/init | 解析漫画目录，生成元数据 |
| 漫画列表 | GET | /api/v1/comic/list | 返回所有漫画基础信息 |
| 漫画详情 | GET | /api/v1/comic/detail | 返回指定漫画详情 |
| 图片列表 | GET | /api/v1/comic/images | 返回漫画所有图片路径 |
| 单张图片 | GET | /api/v1/comic/image | 流式返回单张图片 |
| 进度保存 | PUT | /api/v1/comic/progress | 更新阅读进度 |

#### 2.1.3 接口设计规范
```
请求格式：
- GET: 参数通过 query string 传递
- POST/PUT: 参数通过 JSON body 传递

响应格式：
{
  "code": 200,        // 状态码
  "msg": "成功",      // 提示信息
  "data": {}          // 业务数据
}

错误码：
- 200: 成功
- 400: 参数错误
- 404: 资源不存在
- 500: 服务器内部错误
```

### 2.2 JSON处理模块（json_handler）

#### 2.2.1 模块职责
- 提供JSON文件的读写操作
- 保证数据写入的原子性
- 实现自动备份机制
- 处理文件损坏恢复

#### 2.2.2 核心功能
| 功能 | 方法名 | 输入 | 输出 | 说明 |
|------|--------|------|------|------|
| 读取JSON | read_json() | 文件路径 | Python字典 | 读取并解析JSON文件 |
| 写入JSON | write_json() | 数据、文件路径 | bool | 原子性写入，自动备份 |
| 生成ID | generate_id() | prefix前缀 | string | 生成唯一ID |
| 恢复备份 | restore_backup() | 文件路径 | bool | 从备份恢复数据 |

#### 2.2.3 数据一致性保障
```
写入流程：
1. 读取现有数据
2. 创建备份（.bak文件）
3. 合并新数据
4. 原子性写入
5. 验证写入成功
6. 删除旧备份（可选）

异常处理：
- 写入失败：自动回滚到备份
- 文件损坏：尝试从备份恢复
- 备份损坏：创建新的空数据结构
```

### 2.3 文件解析模块（file_parser）

#### 2.3.1 模块职责
- 扫描漫画图片目录
- 验证图片格式有效性
- 实现自然排序算法
- 返回有序图片列表

#### 2.3.2 核心功能
| 功能 | 方法名 | 输入 | 输出 | 说明 |
|------|--------|------|------|------|
| 解析漫画图片 | parse_comic_images() | comic_id | 图片路径数组 | 扫描目录，返回有序列表 |
| 验证图片格式 | validate_image_format() | 文件路径 | bool | 检查是否为jpg/png/webp |
| 自然排序 | natural_sort() | 文件名列表 | 排序后列表 | 1,2,10正确排序 |

#### 2.3.3 自然排序算法
```
算法思路：
1. 提取文件名中的数字部分
2. 将文件名拆分为 [文本, 数字, 文本, ...] 的数组
3. 对比时数字按数值比较，文本按字符串比较

示例：
["1.jpg", "2.jpg", "10.jpg"] 
→ 正确排序为 ["1.jpg", "2.jpg", "10.jpg"]
而非 ["1.jpg", "10.jpg", "2.jpg"]
```

### 2.4 图片处理模块（image_handler）

#### 2.4.1 模块职责
- 生成封面缩略图
- 流式返回图片资源
- 图片格式转换
- 处理图片损坏情况

#### 2.4.2 核心功能
| 功能 | 方法名 | 输入 | 输出 | 说明 |
|------|--------|------|------|------|
| 生成封面 | generate_cover() | comic_id, 首图路径 | 封面路径 | 压缩生成500px宽缩略图 |
| 获取图片流 | get_image_stream() | comic_id, page_num | 图片流 | 流式返回，避免内存溢出 |
| 创建备份 | create_backup() | 源文件路径 | bool | 创建.bak备份文件 |
| 恢复备份 | restore_backup() | 文件路径 | bool | 从备份恢复 |

#### 2.4.3 封面生成规范
```
处理流程：
1. 读取原始图片
2. 计算缩放比例（目标宽度500px）
3. 保持宽高比缩放
4. 转换格式为JPEG
5. 设置压缩质量85%
6. 保存到 static/cover/{comic_id}.jpg

异常处理：
- 图片损坏：使用默认占位图
- 格式不支持：尝试转换或报错
```

```
                    ┌─────────────┐
                    │   Routes    │

                           │
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
    │json_handler │ │file_parser  │ │image_handler│
    │  (JSON处理)  │ │ (文件解析)  │ │ (图片处理)  │
    └──────┬──────┘ └─────────────┘ └──────┬──────┘
           │                               │
           │         ┌─────────────────────┘
           │         │
    ┌──────▼─────────▼──────┐
    │        Data           │
    │  (文件系统/JSON文件)   │
    └───────────────────────┘
```

### 3.2 接口调用流程

#### 3.2.1 漫画初始化流程
```
[客户端] → POST /api/v1/comic/init
              │
              ▼
[Routes] → 解析参数(comic_id, title)
              │
              ▼
[file_parser] → parse_comic_images(comic_id)
              │ → 扫描目录
              │ → 验证格式
              │ → 自然排序
              │ → 返回图片列表
              │
              ▼
[image_handler] → generate_cover(comic_id, first_image)
              │ → 压缩生成封面
              │ → 返回封面路径
              │
              ▼
[json_handler] → read_json() → 读取现有数据
              │ → 构建新漫画对象
              │ → write_json() → 写入数据
              │
              ▼
[Routes] → 返回响应 {code: 200, data: {...}}
```

#### 3.2.2 图片读取流程
```
[客户端] → GET /api/v1/comic/image?comic_id=xxx&page_num=xxx
              │
              ▼
[Routes] → 解析参数
              │
              ▼
[file_parser] → parse_comic_images(comic_id)
              │ → 获取图片列表
              │ → 验证page_num有效性
              │
              ▼
[image_handler] → get_image_stream(comic_id, page_num)
              │ → 读取图片文件
              │ → 流式返回
              │
              ▼
[Routes] → 返回图片流 (Content-Type: image/jpeg/png/webp)
```

## 4. 可维护性设计

### 4.1 代码组织原则
```
1. 单一职责：每个模块只负责一类功能
2. 依赖倒置：上层依赖下层接口，不依赖具体实现
3. 开闭原则：对扩展开放，对修改关闭
4. 错误隔离：模块内部错误不影响其他模块
```

## 6. 风险规避设计

### 6.1 数据安全风险
| 风险 | 规避措施 |
|------|----------|
| JSON文件损坏 | 自动备份机制，损坏时从备份恢复 |
| 并发写入冲突 | 文件锁机制，保证原子性写入 |
| 磁盘空间不足 | 写入前检查空间，日志记录警告 |
| 误删除数据 | 定期备份，保留历史版本 |

### 6.2 性能风险
| 风险 | 规避措施 |
|------|----------|
| 大图片内存溢出 | 流式读取，不加载完整图片到内存 |
| JSON文件过大 | 按需读取，不加载整个文件到内存 |
| 并发请求过多 | 个人项目并发低，暂不考虑限流 |

### 6.3 异常处理策略
```python
# 统一异常处理框架
try:
    # 业务逻辑
    result = do_something()
    return success_response(result)
except FileNotFoundError as e:
    logger.error(f"文件不存在: {e}")
    return error_response(404, "资源不存在")
except PermissionError as e:
    logger.error(f"权限错误: {e}")
    return error_response(403, "无权限访问")
except Exception as e:
    logger.exception(f"未知错误: {e}")
    return error_response(500, "服务器内部错误")
```
