# 第三方平台完全插件化最终计划书

适用范围：`ULTIMATE_WEB`  
目标：把第三方平台彻底重构为“宿主零平台感知”的插件体系

---

## 1. 最终目标

本计划书对应的最终目标不是“现有协议再优化一点”，而是达到下面这组硬标准：

1. 新增一个第三方平台，在**提供已有功能域能力**时，不修改宿主任何业务代码、前端代码、平台枚举代码、路由分支代码。
2. 平台之间的差异，包括：
   - 配置差异
   - 登录/凭据差异
   - 搜索/详情/收藏/清单/标签/演员等能力差异
   - 资源形态差异
   - 前端展示差异
   - 导入与缓存策略差异
   都下放到插件内部或协议声明中，不允许继续散落在宿主代码里。
3. 宿主不再识别“JM / PK / JAVDB / JAVBUS / Missav”这些平台名，只识别：
   - 插件
   - 能力
   - 协议字段
4. 前端不再通过平台名写死适配，例如：
   - `javdb` 用 `16/9`
   - `javbus` 用 `2/3`
   这类差异必须由插件协议提供。
5. 第三方库内部允许并且应该保留对其原始实现的耦合，但这种耦合必须被限制在插件目录内部，不能泄漏到宿主。
6. `comic_backend/third_party/legacy` 最终应被清空并删除。

补充说明：

1. “新增平台零宿主改动”只适用于“接入已有功能域”。
2. 如果新增的是全新的产品功能域，例如宿主从来没有“弹幕下载”“字幕翻译”“种子做种管理”这类功能，那么宿主仍然需要新增通用功能模块。
3. 但即便新增功能域，也必须一次性做成**协议化通用能力**，而不是为某个平台加一套特判。

## 3. 为达到最终目标，协议必须升级到 v2

当前协议 `v1.0` 只解决了：

1. 插件发现
2. 能力声明
3. 配置 schema
4. provider 执行入口

这还不够。

要达到“新增平台零宿主改动”，协议必须覆盖宿主今天所有仍在硬编码的平台差异。

### 3.1 协议 v2 必须新增的 8 类信息

#### 1. 身份协议 `identity`

宿主不能再依赖静态 `Platform` 枚举和固定前缀。

插件必须声明：

1. `plugin_id`
2. `content_domain`
3. `entity_kind`
4. `host_id_policy`
5. `source_id_policy`
6. `display_code_policy`

建议字段：

```json
{
  "identity": {
    "content_type": "video",
    "host_id_prefix": "JAVDB",
    "host_id_format": "{prefix}{remote_id}",
    "remote_id_field": "video_id",
    "display_code_field": "code"
  }
}
```

进一步建议：

1. 长期最好把宿主主键升级为 `plugin_id + remote_id` 的组合身份，而不是继续依赖固定前缀。
2. 如果短期要兼容旧数据，可以让前缀也由插件 manifest 声明，而不是宿主枚举声明。

#### 2. 展示协议 `presentation`

这是实现你提出的“封面长宽比由插件告诉前端”的核心。

插件必须能声明：

1. 卡片封面长宽比
2. 封面填充方式 `cover/contain`
3. 默认卡片布局
4. 标题行数
5. 是否展示平台角标
6. 平台角标文案
7. 是否允许预览视频
8. 是否要求封面代理
9. 订阅页/搜索页/清单页的默认排序建议

建议字段：

```json
{
  "presentation": {
    "media_card": {
      "cover": {
        "aspect_ratio": "16/9",
        "fit": "cover"
      },
      "badge": {
        "show_platform_label": true,
        "label": "JAVDB"
      }
    },
    "search": {
      "default_sort_weight": 100
    },
    "import": {
      "allow_preview_video": true
    }
  }
}
```

补充要求：

1. 既要支持插件级默认展示协议。
2. 也要支持单条记录在返回结果里覆盖展示协议，例如同一插件下不同内容源的图片比例不同。

#### 3. 能力参数协议 `capability contract`

当前只有 capability key，没有严格的输入输出约定。

必须补充：

1. `request_schema`
2. `response_schema`
3. `result_model`
4. `supported_modes`
5. `requires_auth`
6. `supports_pagination`
7. `supports_import_target`

否则宿主还是会继续写平台特判。

#### 4. 交互动作协议 `actions`

当前 `helper_urls` 太弱，只能支持简单帮助链接。

要升级成通用交互动作：

1. 打开帮助页
2. 打开登录页
3. 复制说明文案
4. 校验凭据
5. 刷新会话
6. 探测标签库
7. 探测在线播放能力

建议字段：

```json
{
  "actions": [
    {
      "key": "auth.open_help",
      "label": "打开 Cookie 获取教程",
      "kind": "open_url",
      "url": "/api/v1/plugin/video.javdb/action/auth.open_help"
    }
  ]
}
```

#### 5. 资源策略协议 `resource policy`

今天宿主仍然自己决定：

1. 封面存放到哪个目录
2. 是否走图片代理
3. 预览视频是否缓存
4. 哪些源需要特殊请求头

这些都应该变成插件协议的一部分。

插件必须能声明：

1. `cover_proxy_mode`
2. `preview_video_policy`
3. `download_asset_policy`
4. `request_header_policy`
5. `local_cache_policy`

#### 6. 列表/收藏/清单语义协议 `collection semantics`

今天宿主里存在：

1. JM/PK 收藏夹是轻量漫画列表
2. JAVDB 收藏夹/清单语义不同
3. 某些平台支持虚拟 favorites，某些不支持

必须协议化：

1. 是否支持 favorites
2. 是否支持 favorites_basic
3. 是否支持 custom_lists
4. 是否支持 list detail
5. list item 的返回模型
6. 同步策略与唯一键策略

#### 7. 人物/作者/标签协议 `taxonomy & people`

今天视频与漫画两条链路的人物和标签能力都很碎，且平台差异明显。

需要统一表达：

1. 是否支持 `person.search`
2. 是否支持 `person.works`
3. 是否支持 `taxonomy.tags`
4. 是否支持 `taxonomy.tree`
5. 是否支持 `taxonomy.tag_search`
6. 是否支持本地缓存标签树

尤其像 `javdb tags` 这种当前专用接口，未来必须升级成通用 taxonomy 协议。

#### 8. 错误与健康协议 `status & errors`

插件必须统一输出：

1. `configured`
2. `message`
3. `missing_fields`
4. `health_state`
5. `degraded_capabilities`
6. `retryable`

宿主不应该继续自己猜哪些平台缺 cookie、哪些平台没账号。

## 4. 最终版目标架构

### 4.1 宿主最终只保留 7 类模块

1. `PluginRegistry`
2. `ProviderManager`
3. `CapabilityRouter`
4. `PluginConfigCenter`
5. `PluginPresentationResolver`
6. `PluginTaskDispatcher`
7. `PluginResultNormalizer`

宿主最终不应该再保留：

1. `Platform` 静态枚举
2. 平台到插件映射表
3. 平台专用 API 分支
4. 平台专用前端样式分支
5. `third_party/legacy`

### 4.2 插件最终目录形态

建议最终每个第三方平台都变成完整插件包：

```text
comic_backend/third_party/<plugin-name>/
  ultimate-plugin.json
  ultimate_provider.py
  translators/
  resources/
  adapters/
  tests/
  vendor_code/
```

说明：

1. `vendor_code/` 或 `adapters/` 中保留插件对原始第三方实现的耦合。
2. 这种耦合是允许的。
3. 但它必须被封装在插件目录里，不能再散落到宿主。

### 4.3 宿主与插件的唯一交互方式

最终宿主只允许以下几类交互：

1. 读取 manifest
2. 读取插件配置 schema
3. 查询插件健康状态
4. 执行 capability
5. 执行 action
6. 读取展示协议

除这几类交互之外，宿主不得再 import 平台实现类，不得再写平台名分支。

## 5. 必须完成的改造项

### 5.1 后端必须完成

#### 1. 删除静态平台枚举依赖

当前：

1. [core/platform.py](D:/code/临时并行开发目录/ULTIMATE_WEB/comic_backend/core/platform.py)
2. 大量业务服务依赖 `Platform.JM / Platform.PK / Platform.JAVDB / Platform.JAVBUS`

最终要求：

1. 宿主以 `plugin_id` 或 manifest 驱动 identity 运行。
2. 现有 `Platform` 只允许作为迁移期兼容层存在。
3. 最终业务层不再直接依赖 `Platform`。

#### 2. 删除 `protocol/compatibility.py` 中静态映射

当前：

1. `COMIC_PLATFORM_PLUGIN_MAP`
2. `GENERAL_PLATFORM_PLUGIN_MAP`
3. `VIDEO_PLATFORM_PLUGIN_MAP`
4. `ADAPTER_PLUGIN_MAP`

最终要求：

1. 插件路由必须由 manifest 动态查询。
2. 不能继续依赖写死的映射表。

#### 3. 把平台专用能力改成通用 capability

必须改掉的典型接口：

1. `javdb cookie-status`
2. `javdb tags`
3. `javdb search-by-tags`
4. `missav playback`
5. `javbus preview policy`

最终要求：

1. 宿主只调用通用 capability，比如：
   - `auth.status`
   - `taxonomy.tags`
   - `taxonomy.search`
   - `playback.sources.build`
   - `playback.proxy.stream`
2. 哪个插件支持，由插件 manifest 决定。

#### 4. 把配置帮助入口变成 action

当前：

1. `javdb_cookie_guide` 是平台专用帮助入口。

最终要求：

1. 所有配置帮助、登录帮助、检测动作都通过插件 actions 下发。
2. 系统配置页不再识别 `adapterName === 'javdb'`。

#### 5. 把资源策略协议化

当前宿主仍然写死：

1. 封面目录
2. 预览缓存目录
3. 资源代理策略
4. 特殊请求头拼装

最终要求：

1. 宿主只负责执行资源策略。
2. 资源策略的差异由插件协议提供。

### 5.2 前端必须完成

#### 1. 删除平台名驱动的卡片样式

当前典型文件：

1. [MediaGrid.vue](D:/code/临时并行开发目录/ULTIMATE_WEB/comic_frontend/src/components/common/MediaGrid.vue)
2. [GlobalSearch.vue](D:/code/临时并行开发目录/ULTIMATE_WEB/comic_frontend/src/views/search/GlobalSearch.vue)
3. [CreatorDetail.vue](D:/code/临时并行开发目录/ULTIMATE_WEB/comic_frontend/src/views/subscribe/CreatorDetail.vue)

最终要求：

1. 前端根据 `item.display` 或 `plugin.presentation` 渲染。
2. 前端不得再判断 `javdb/javbus/JM/PK`。

#### 2. 删除平台专用远程搜索 UI

当前：

1. 视频远程搜索默认平台 `JAVDB`
2. 漫画远程搜索默认平台 `JM`
3. 选中导入时按平台分组

最终要求：

1. 搜索页面从宿主接口拿到“当前支持该 capability 的插件列表”。
2. 页面按插件协议渲染筛选项。

#### 3. 删除配置页平台专用渲染

当前：

1. `SystemConfig.vue` 有 `javdb` 特判 UI。

最终要求：

1. 完全根据插件配置 schema + actions 渲染。
2. 不允许再通过平台名判断。

#### 4. 删除订阅页/搜索页的平台检测器

当前：

1. 通过 `platform` / `id` / `cover_url` 猜测 `javdb/javbus`

最终要求：

1. 记录里直接携带统一的 `plugin_id`
2. 或携带 `display.profile`
3. 前端不得再靠字符串猜平台。

## 7. 协议 v2 示例：封面比例问题怎么解决

以你举的 `JAVDB / JAVBUS` 封面比例不同为例，最终不应该这样做：

1. 前端判断 `if platform === 'javdb'`
2. 前端判断 `if platform === 'javbus'`

而应该这样做：

### 插件 manifest

```json
{
  "presentation": {
    "media_card": {
      "cover": {
        "aspect_ratio": "16/9",
        "fit": "cover"
      }
    }
  }
}
```

或者如果需要按单条记录覆盖：

### 插件结果

```json
{
  "items": [
    {
      "id": "video.javdb:ABP-123",
      "title": "ABP-123",
      "cover_url": "https://...",
      "display": {
        "cover": {
          "aspect_ratio": "16/9",
          "fit": "cover"
        }
      }
    }
  ]
}
```

前端统一做法：

1. 先读 item 级 `display`
2. 没有则读 plugin 级 `presentation`
3. 没有则退回宿主默认值

这样前端永远不用知道平台名。

## 8. 当前文件夹的最终去留判断

### 8.1 `comic_backend/third_party/adapter*.py` 能否删除

当前结论：**现在不能全删，最终应该删掉。**

原因：

1. 现在很多 provider 还通过这些兼容壳拿旧实现。
2. 当前 `runtime_config` 还包着旧 `AdapterConfig`。
3. 当前 `MetaDataAdapter` 这类转换器仍然在宿主业务里使用。

最终状态：

1. 宿主侧 adapter 文件全部删除。
2. 通用转换器若确实与宿主无关，迁到插件内部。
3. 真正宿主公共逻辑迁到 `protocol/` 或 `application/core`。

### 8.2 `comic_backend/third_party/legacy` 是什么

当前结论：**它是迁移隔离区，不是最终设计。**

它的作用只有一个：

1. 在不破坏现有功能的前提下，把旧适配实现先隔离起来，给协议化迁移留缓冲层。

最终状态：

1. `legacy` 必须清空并删除。

### 8.3 能否做到“软件与第三方平台完全解耦”

答案：

1. **宿主与第三方平台完全解耦**，可以做到。
2. **插件内部与第三方原始实现完全无耦合**，没有必要，也不现实。

最终正确边界应该是：

1. 宿主只认协议。
2. 插件内部消化全部第三方差异。
3. 第三方实现怎么变，由插件自己兜底。

## 9. 最终验收标准

只有同时满足下面 10 条，才算真正达到目标：

1. 新增一个已有功能平台时，不改宿主后端代码。
2. 新增一个已有功能平台时，不改宿主前端代码。
3. 新增一个已有功能平台时，不改 `Platform` 枚举或任何平台映射表。
4. 前端不再根据平台名判断长宽比、封面填充、排序、导入行为。
5. 宿主后端不再存在 `if platform == ...` 的平台特判分支。
6. 宿主不再存在 `javdb tags` 这类平台专用能力接口。
7. `comic_backend/third_party/legacy` 被删除。
8. `comic_backend/third_party` 根目录不再存在宿主侧 adapter 实现。
9. 现有全部第三方集成测试通过。
10. 新增一个 demo 插件时，只需新增插件目录和测试，不改宿主。

## 10. 建议执行顺序

建议严格按下面顺序推进：

1. 先做“耦合清单冻结”
2. 再做“协议 v2 冻结”
3. 再做“宿主身份系统动态化”
4. 再做“前端展示协议化”
5. 再做“现有插件内部重构”
6. 最后删掉 `legacy` 和所有兼容壳

原因很简单：

1. 如果不先补协议，后面只会把平台特判从一个文件搬到另一个文件。
2. 如果不先做前端协议化，后端再干净，新增平台还是会卡在 UI 上。
3. 如果不先把测试改成协议门禁，删 `legacy` 的时候风险会很大。
