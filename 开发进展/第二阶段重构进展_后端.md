# 第二阶段重构进展 - 后端部分

## 一、整体架构

后端采用 **Flask + 分层架构**，实现了漫画管理、标签管理、图片服务等功能。

### 技术栈
- **Flask**: Web 框架
- **PIL (Pillow)**: 图片处理
- **JSON 文件**: 数据持久化

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                        接口层 (API)                          │
│  - 职责：HTTP 请求处理、参数校验、响应封装                      │
│  - 原则：不包含业务逻辑，只负责输入输出转换                      │
├─────────────────────────────────────────────────────────────┤
│                        应用层 (Application)                  │
│  - 职责：业务流程编排、事务管理                                │
│  - 原则：协调多个领域服务完成业务用例                           │
├─────────────────────────────────────────────────────────────┤
│                        领域层 (Domain)                       │
│  ┌─────────────┐ ┌─────────────┐                            │
│  │  Comic领域  │ │   Tag领域   │                            │
│  │  - Entity   │ │  - Entity   │                            │
│  │  - Repository│ │  - Repository│                           │
│  └─────────────┘ └─────────────┘                            │
├─────────────────────────────────────────────────────────────┤
│                        基础设施层 (Infrastructure)           │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────────┐       │
│  │  JSON存储   │ │   日志系统   │ │   通用工具      │       │
│  │  - 持久化   │ │  - 日志记录  │ │  - 返回结果     │       │
│  │  - 缓存     │ │             │ │  - 异常处理     │       │
│  └─────────────┘ └─────────────┘ └─────────────────┘       │
├─────────────────────────────────────────────────────────────┤
│                        核心层 (Core)                         │
│  - 常量定义、枚举定义、通用工具函数                            │
└─────────────────────────────────────────────────────────────┘
```

## 四、架构优势

### 4.1 分层清晰
- **API 层**：只负责 HTTP 请求处理
- **Application 层**：业务流程编排
- **Domain 层**：核心业务逻辑
- **Infrastructure 层**：技术实现细节

### 4.2 易于扩展
- 新增功能只需添加新的应用服务和接口
- 仓储接口支持多种实现（JSON、数据库等）
- 预留了导入策略、图床代理等扩展点

### 4.3 便于测试
- 每层职责单一，易于单元测试
- 依赖注入，方便 Mock

### 4.4 可维护性
- 代码结构清晰，易于理解
- 统一的返回结果和异常处理
- 完善的日志记录
