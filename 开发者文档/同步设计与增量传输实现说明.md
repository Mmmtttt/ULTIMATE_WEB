# 同步设计与增量传输实现说明（后端现状）

## 1. 文档目标

本文档描述当前后端同步能力的完整实现，包括：

1. 会话式同步（`session/manifest/download/finish`）
2. 节点间定向同步（`pairing + directional push/pull`）
3. 元数据与资源的去重/增量算法
4. `tag/list` 同 ID 冲突修复算法（含引用回写）
5. 实时进度任务模型

本文是“代码实现说明”，以 `comic_backend` 代码为准。

## 4. 两条同步链路

## 4.1 会话式同步（Session）

用于“打包元数据给客户端下载”的同步模式。

### 4.1.1 接口

- `POST /api/v1/sync/session`
- `GET /api/v1/sync/manifest/<session_id>`
- `GET /api/v1/sync/download/<session_id>/<package_name>`
- `POST /api/v1/sync/session/finish`
- `GET /api/v1/sync/session/<session_id>`

### 4.1.2 实现特征

- 会话存储文件：`meta_data/sync_sessions.json`
- 会话导出目录：`cache/sync_exports/<session_id>/`
- 当前会话包：`meta_data.zip`
- 默认 TTL：24 小时
- 包清单包含 `size_bytes` 与 `sha256`

说明：当前会话模式主要用于元数据导出，不包含资源目录增量逻辑。

## 4.2 定向同步（Directional）

用于两个后端节点间双向同步，支持 `push` 和 `pull`。

### 4.2.1 接口分组

1. 配对与节点管理
- `POST /api/v1/sync/pairing/invite`
- `POST /api/v1/sync/pairing/claim`
- `POST /api/v1/sync/pairing/connect`
- `GET /api/v1/sync/peers`
- `DELETE /api/v1/sync/peers/<peer_id>`

2. 元数据增量
- `GET /api/v1/sync/directional/inventory`
- `POST /api/v1/sync/directional/delta`
- `POST /api/v1/sync/directional/apply`
- `POST /api/v1/sync/directional/estimate`
- `POST /api/v1/sync/directional/preview`

3. 资源增量
- `GET /api/v1/sync/directional/assets/inventory`
- `POST /api/v1/sync/directional/assets/apply`
- `GET|POST /api/v1/sync/directional/assets/delta/download`

4. 执行入口
- `POST /api/v1/sync/directional/push`
- `POST /api/v1/sync/directional/pull`

5. 异步任务与进度
- `POST /api/v1/sync/directional/task/start`
- `GET /api/v1/sync/directional/task/<task_id>`

## 5. 元数据算法（Directional）

## 5.1 数据集定义

`DirectionalSyncService.DATASETS` 当前包含：

- `comics`
- `recommendations`
- `videos`
- `video_recommendations`
- `tags`
- `lists`
- `actors`
- `authors`
- `user_config`（字典类型）

## 5.2 去重标准（元数据）

元数据的“记录重复”判定标准：

- 标准键：`<dataset_path>::<id>`
- 即：同一数据集路径下，`id` 相同即视为同一条记录

实现点：

- 增量构建时使用数据集 + ID 去重
- 应用增量时也会过滤同一批次重复 ID

## 5.3 常规数据集增量规则

适用：`comics/recommendations/videos/video_recommendations/actors/authors`

规则：

1. 读取本地数据集
2. 读取对端 inventory 的 `ids`
3. 发送“本地有、对端没有”的记录
4. 接收端按非破坏式合并写入

## 5.4 `tags/lists` 语义键与冲突修复

这是本次重点修复内容，解决“同 ID 不同内容”。

### 5.4.1 语义键定义

1. Tag 语义键
- `content_type + name(casefold + trim)`
- 形式：`<content_type>|<normalized_name>`

2. List 语义键
- 远程跟踪清单优先：`remote|content_type|platform|platform_list_id|import_source`
- 普通清单回退：`local|content_type|name(casefold + trim)`

### 5.4.2 Inventory 扩展

`tags/lists` 在 inventory 中除了 `ids`，还会额外提供 `keys`（语义键集合）。

### 5.4.3 Delta 发送策略

`tags/lists` 的发送规则：

1. 若对端提供 `keys`：按语义键差集发送
2. 若对端未提供 `keys`（旧版本兼容）：发送完整 `tags/lists` 数据集

这样可以覆盖“同 ID 不同内容”的场景，避免被纯 ID 差集误判为“已同步”。

### 5.4.4 Apply 冲突处理

在 `apply_delta()` 正式写库前，先执行 ID 冲突重映射：

1. 构建本地 `tags/lists` 的
- `existing_by_id`
- `semantic_key -> existing_id`

2. 遍历 incoming 的 `tags/lists`：
- ID 存在且语义相同：保留 ID
- ID 存在但语义不同：判定为冲突
- 若本地存在相同语义：映射到本地已有 ID
- 若本地不存在相同语义：生成新 ID（`tag_sync_*` / `list_sync_*`）

3. 形成 `id_map` 后，回写所有 incoming 数据集中的引用字段：
- `tag_ids`
- `list_ids`

4. 最终再进入 `_apply_dataset()` 原子落盘

结果：即使出现“同 ID 不同内容”，也不会覆盖错对象，且内容记录中的关联关系会自动修正到新 ID。

## 5.5 接收端写入策略（非破坏式）

`_apply_dataset()` + `_merge_row()` 的核心规则：

1. 新 ID：新增记录
2. 已存在 ID：仅补齐空字段，不覆盖已有非空字段
3. 列表字段（`tag_ids/list_ids/actors`）：并集去重
4. `user_config`：递归缺失字段补齐，保留本地已有值

## 6. 资源算法（Directional）

## 6.1 去重标准（资源文件）

资源文件的“重复”判定标准：

- 标准键：文件相对于 `DATA_DIR` 的路径
- 例如：
  - `.../data/recommendation_cache/video/JAVDB/JAVDB0RE39E`
  - 与任意根目录下同相对路径的文件
  - 视为同一资源

实现点：

1. 路径统一标准化为 `/` 分隔
2. 去除前导 `/`
3. 仅比较相对路径，不再依赖 `mtime` 差异判重

## 6.2 资源索引

`asset_inventory()` 仍输出：

- `files: { rel_path -> "size:mtime" }`

说明：`size:mtime` 主要用于估算字节规模与展示，不作为重复判定唯一依据。

## 6.3 资源增量判定

### 6.3.1 Push

发送端构建本地索引后：

- 若 `rel_path` 在对端索引键中存在 => 视为已存在，跳过
- 若不存在 => 纳入增量 zip

### 6.3.2 Pull

拉取端把本地路径键发送给对端：

- 对端只打包本端存在且拉取端不存在的 `rel_path`

## 6.4 资源传输与应用

1. Push 上传：`assets/apply`（multipart zip）
2. Pull 下载：`assets/delta/download`（zip stream）
3. 解压白名单目录：`comic/ video/ static/ cache/ recommendation_cache/`
4. 防穿越校验：拒绝 `../` 与越界路径

## 7. 任务化与实时进度

## 7.1 任务模型

任务结构包含：

- `task_id`
- `peer_id`
- `direction`
- `status`（queued/running/completed/failed）
- `stage`
- `progress`
- `message`
- `extra`
- `result`
- `error`
- `created_at/started_at/updated_at/finished_at`

任务保存在进程内存（`_TASKS`），默认最多保留 50 条。

## 7.2 典型阶段

1. `prepare`

5. `data_apply_remote`

7. `asset_push_upload`
8. `completed`

Pull：

1. `prepare`
2. `local_inventory`
3. `remote_delta`
4. `data_apply_local`
5. `asset_pull_request`
6. `asset_pull_download`
7. `asset_pull_extract`
8. `completed`

## 8. 兼容性与边界

1. 对端不支持资源端点（404/405）时：
- 返回 `unsupported_remote`
- 元数据同步继续执行

2. 任务进度为工程化进度，不是严格字节总进度。

3. 任务状态为内存态，服务重启后历史任务不保留。

4. 与旧版本 peer 的兼容：
- `pull assets` 遇到 405 会回退尝试 GET
- `tags/lists` 若对端 inventory 无 `keys`，会走全量发送以保障冲突修复
