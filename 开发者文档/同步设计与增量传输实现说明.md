# 同步设计与增量传输实现说明

## 1. 目标与范围

本同步系统用于在两个后端节点之间进行双向定向同步（`push` / `pull`），同步对象包括：
- 元数据数据库（JSON 数据库）
- 媒体与资源文件（图片、视频、封面、缓存、推荐缓存）

本次改动新增了**实时进度能力**，前端可看到同步过程中的阶段与百分比。

## 4. 增量算法原理

## 4.1 数据库增量（JSON）

核心思想：**按数据集 + 主键 ID 计算缺失项增量**，不是全量覆盖。

关键函数：
- `DirectionalSyncService.inventory()`
- `DirectionalSyncService.delta_from_known()`
- `DirectionalSyncService.apply_delta()`
- `DirectionalSyncService._apply_dataset()`
- `DirectionalSyncService._merge_row()`

### 4.1.1 发送端如何算“我该发什么”

对每个数据集（如 comics/tags/videos）：
1. 读取本地列表
2. 读取对端 inventory 的 id 集
3. 发送 `本地有、对端没有` 的记录

对 `user_config`（字典型）：
- 按“缺失字段补齐”策略增量合并，不做粗暴覆盖。

### 4.1.2 接收端如何合并

不是直接替换整表，使用“非破坏式合并”：
- `id` 不存在：新增记录
- `id` 已存在：仅补齐缺失字段
- 列表字段（`tag_ids/list_ids/actors`）做并集去重
- 避免把对端已有字段覆盖成空

这保证了 `A={1,2,3}` 推到 `B={2,3,4}` 后可得到并集语义（并保持已有信息不丢失）。

---

## 4.2 资产增量（图片/视频/缓存）

核心思想：**文件签名对比 + 仅传差异文件**。

关键函数：
- `DirectionalSyncService._collect_asset_index()`
- `DirectionalSyncService.build_asset_delta_zip()`
- `DirectionalSyncService._create_asset_zip()`
- `DirectionalSyncService._extract_asset_zip()`

### 4.2.1 文件签名

每个文件签名格式：
- `"{size}:{mtime}"`

其中：
- `size`：文件字节大小
- `mtime`：修改时间（秒级）

发送端与接收端各自产生 `rel_path -> signature` 索引。

### 4.2.2 增量判断

当满足任一条件即视为增量文件：
- 对端不存在该路径
- 对端同路径签名不同（大小或修改时间变化）

### 4.2.3 传输方式

`push`：发送端将增量文件打包 zip，通过 `assets/apply` 上传，接收端解压覆盖对应路径。  
`pull`：接收端把本地索引发给对端，对端返回增量 zip，接收端下载后解压应用。

## 5. 本次新增：实时进度条

## 5.1 后端任务化模型

新增“同步任务”机制，把原本阻塞式 `push/pull` 包装成可查询状态任务。

关键新增函数（`sync_directional_service.py`）：
- `start_directional_task(peer_id, direction)`
- `get_directional_task(task_id)`
- `_execute_directional_task(task_id)`
- `_update_directional_task(...)`
- `_report_progress(...)`

任务状态字段包含：
- `task_id`
- `peer_id`
- `direction`
- `status`（queued/running/completed/failed）
- `stage`（当前阶段）
- `progress`（0-100）
- `message`
- `extra`（扩展统计：记录数、文件数、字节数等）
- `result` / `error`
- `created_at/started_at/updated_at/finished_at`

### 5.1.1 新增 API

文件：`comic_backend/api/v1/sync.py`

- `POST /api/v1/sync/directional/task/start`
  - 入参：`peer_id`, `direction(push|pull)`
  - 返回：任务快照（含 `task_id`）

- `GET /api/v1/sync/directional/task/<task_id>`
  - 返回：最新任务状态（可轮询）

## 5.2 进度阶段设计（示例）

`push` 典型阶段：
1. prepare
2. remote_inventory
3. data_delta
4. asset_inventory
5. data_apply_remote
6. asset_push / asset_push_upload
7. completed

`pull` 典型阶段：
1. prepare
2. local_inventory
3. remote_delta
4. data_apply_local
5. asset_pull_request
6. asset_pull_download
7. asset_pull_extract
8. completed

注：资产下载与解压阶段会持续刷新进度，属于“实时变化”区段。

## 7. 为什么这是“增量传输”

系统并非每次全量传所有文件/所有记录，而是：
- 数据库：按 ID 差异传输缺失记录，接收端按字段级规则合并
- 资产：按文件签名差异仅打包变更文件

因此在多次同步中，后续传输量会显著下降，网络与 IO 成本更低。

---

## 8. 兼容性与边界

- 如果对端不支持资产端点（404/405），会返回 `unsupported_remote`，数据库同步仍可继续。
- 任务状态保存在进程内存中，服务重启后历史任务会丢失（当前设计用于实时进度展示，不作为持久审计系统）。
- 进度值是“阶段+子进度”的工程化进度，不等于严格字节级总进度（但资产下载/解压阶段具备字节/文件推进）。
