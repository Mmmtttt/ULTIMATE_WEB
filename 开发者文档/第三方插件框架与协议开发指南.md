# 第三方插件框架与协议开发指南

更新时间：2026-04-24  
适用版本：当前 `ULTIMATE_WEB` 工作区实现

---

## 1. 文档定位

这份文档描述的是“当前代码已经落地的第三方插件框架”，重点回答下面几件事：

1. 宿主现在如何读取第三方库信息。
2. 宿主现在如何配置第三方库参数。
3. 代码里如何调用第三方库能力。
4. 当前协议 `ultimate-plugin.json` 的格式和含义是什么。
5. 当前已经有哪些插件、有哪些能力。
6. 如果要新增一个平台，复用现有功能，应该怎么做。
7. 如果要扩展一个全新的功能域，应该怎么设计，才能继续保持“宿主面向协议，而不是面向平台”。

这份文档描述的是“现状实现”，不是早期设计草案。  
设计阶段文档仍可参考，但以当前代码为准。

---

## 2. 设计目标与当前结论

当前第三方体系的核心原则是：

1. 宿主不再直接面向平台实现，而是面向插件协议。
2. 平台差异优先写入插件自身：
   - 配置差异
   - 展示差异
   - 资源请求差异
   - 登录凭据差异
   - 列表/收藏/标签/演员等能力差异
3. 宿主只负责：
   - 扫描插件
   - 校验 manifest
   - 加载 provider
   - 按 capability 调用
   - 根据协议字段做通用展示和通用调度
4. 如果某个平台有特殊行为，优先通过协议字段表达；不要在宿主继续写 `if platform == "javdb"` 这种分支。

一个最典型的例子就是封面比例：

- `JAVDB` 视频封面应该是横版，已经通过 `presentation.media_card.cover.aspect_ratio = "16 / 9"` 声明。
- `JAVBUS` 视频封面应该是竖版，已经通过 `presentation.media_card.cover.aspect_ratio = "2 / 3"` 声明。

前端不需要再识别平台名，而是直接读取协议里的展示信息。

## 3. 当前目录与模块分工

### 3.1 插件目录

宿主会递归扫描：

```text
comic_backend/third_party/**/ultimate-plugin.json
```

也就是说：

1. 一个第三方库可以直接就是一个插件目录。
2. 一个第三方库内部也可以再拆出多个插件。
3. 插件不要求都在一级目录。

当前插件分布：

```text
comic_backend/third_party/JMComic-Crawler-Python/ultimate-plugin.json
comic_backend/third_party/Picacomic-Crawler/ultimate-plugin.json
comic_backend/third_party/javdb-api-scraper/ultimate-plugin.json
comic_backend/third_party/javdb-api-scraper/javbus_plugin/ultimate-plugin.json
comic_backend/third_party/Missav/ultimate-plugin.json
```

### 3.2 宿主协议层

当前协议层核心模块：

| 模块 | 作用 |
| --- | --- |
| `comic_backend/protocol/registry.py` | 扫描并校验 `ultimate-plugin.json`，构建 `PluginManifest` |
| `comic_backend/protocol/base.py` | 定义 `PluginManifest`、`ProtocolProvider`、`ProtocolProviderClient` |
| `comic_backend/protocol/provider_manager.py` | 根据 `plugin.entrypoint` 动态加载 provider 类，并注入配置 |
| `comic_backend/protocol/gateway.py` | 宿主统一入口，按 `plugin_id + capability` 执行插件 |
| `comic_backend/protocol/runtime_config.py` | 读写 `third_party_config.json`，并把 manifest 默认值合入运行时配置 |
| `comic_backend/protocol/config_service.py` | 为前端系统配置页生成动态 schema 与当前配置 |
| `comic_backend/protocol/presentation.py` | 把 manifest 里的展示信息合并到返回项的 `display` 字段 |
| `comic_backend/protocol/platform_meta.py` | 处理平台查找、前缀 ID、默认平台、host 前缀等通用元信息 |
| `comic_backend/protocol/adapter_api.py` | 面向旧漫画适配接口的协议封装 |
| `comic_backend/protocol/platform_service.py` | 面向旧平台服务接口的协议封装 |

### 3.3 当前仍保留的兼容壳

下面这些文件仍然存在，但它们本质上只是协议层别名或包装，不是新的主架构：

| 文件 | 当前角色 |
| --- | --- |
| `comic_backend/third_party/external_api.py` | 兼容旧导入路径，实际转发到 `protocol.adapter_api` |
| `comic_backend/third_party/platform_service.py` | 兼容旧导入路径，实际转发到 `protocol.platform_service` |
| `comic_backend/application/platform_service.py` | 应用层兼容包装，内部还是走协议服务 |

新代码应优先直接使用协议层，不应再围绕这些兼容壳继续设计新逻辑。

## 4. 宿主如何读取第三方库信息

### 4.1 扫描与校验

启动后，宿主通过 `PluginRegistry` 递归扫描 `comic_backend/third_party`，找到所有 `ultimate-plugin.json`。

当前校验规则：

1. `protocol_version` 必须是 `1.0`、`1.1` 或 `2.0` 之一。
2. `plugin.id` 必填。
3. `plugin.entrypoint` 必填。

扫描后的每个 manifest 会被包装成 `PluginManifest` 对象。

### 4.2 宿主如何识别一个插件

宿主并不是只认一个字段，而是会构造一组“查找名”：

- `plugin_id`
- `config_key`
- `plugin.name`
- `identity.platform_label`
- `identity.host_id_prefix`
- `identity.aliases`

也就是说，下面这些名字都可以用于查找同一个插件：

- `javdb`
- `JAVDB`
- `video.javdb`
- manifest 中声明的别名

对应实现见：

- `PluginManifest.list_lookup_names()`
- `PluginRegistry.find_by_lookup_name()`

### 4.3 宿主如何读取展示差异

宿主会读取 manifest 中的 `presentation` 字段，并通过 `protocol.presentation.annotate_item()` / `annotate_items()` 把结果合并到每一条返回项的 `display` 中。

这意味着：

1. 前端卡片长宽比可以来自协议。
2. 平台角标可以来自协议。
3. 封面显示策略可以来自协议。

宿主不需要再写平台分支。

## 5. 宿主如何配置第三方库参数

### 5.1 配置文件来源

插件运行时配置统一存放在：

```text
third_party_config.json
```

这个文件的真实路径由 `comic_backend/core/constants.py` 动态解析，最终由 `THIRD_PARTY_CONFIG_PATH` 指向。

### 5.2 配置加载过程

配置流程如下：

1. `ProtocolConfigStore` 读取 `third_party_config.json`。
2. 遍历所有 manifest，把每个插件的默认配置合并进去。
3. 如果 manifest 声明了 `storage.data_dir_bindings`，会自动补齐默认目录。
4. 如果用户保存了配置，则以用户配置覆盖默认值。
5. 配置更新后会重置相关 provider 缓存，保证新配置立即生效。

### 5.3 前端配置页如何生成

前端系统设置页不是写死字段，而是这样生成的：

1. 前端请求 `GET /api/v1/comic/third-party/config`
2. 后端 `PluginConfigService.build_response()`：
   - 读取所有 manifest
   - 过滤出“有 `config_key` 且存在配置字段”的插件
   - 按 `configuration.sections[].fields[]` 生成 schema
   - 读取当前配置值
   - 调用 provider 的 `serialize_public_config()` 做脱敏/回显转换
3. 前端 `SystemConfig.vue` 根据返回的 `schema` 渲染表单

### 5.4 一个配置接口返回什么

当前接口返回的核心字段有：

| 字段 | 说明 |
| --- | --- |
| `default_adapter` | 默认配置键 |
| `adapter_order` | 配置面板展示顺序 |
| `schema` | 动态表单 schema |
| `adapters` | 当前各插件配置值 |
| `helper_urls` | 配置辅助页地址 |
| `plugins` | 所有插件的公开描述 |
| `configurable_plugins` | 可配置插件的公开描述 |

### 5.5 helper 页面与交互配置

协议支持插件为自己的配置项提供帮助页。

当前已落地的实现：

1. manifest 里的 `helpers` 定义辅助资源。
2. `configuration.actions` 定义配置页按钮。
3. 当前支持的 helper 类型是 `static_page`。
4. 当前支持的配置页动作主要是 `open_url`。

例如 `JAVDB` 使用这个机制提供 Cookie 获取教程页。

## 6. 宿主如何调用第三方库能力

### 6.1 总体链路

宿主现在的调用方式是：

```text
业务代码 -> ProtocolGateway -> ProviderManager -> ProtocolProvider.execute() -> 第三方库
```

更完整一点：

```text
API / Application Service
  -> 根据平台名或 capability 查 manifest
  -> ProtocolGateway.execute_plugin(plugin_id, capability, params)
  -> ProviderManager.get_provider(plugin_id)
  -> 按 plugin.entrypoint 动态加载 provider 类
  -> ProviderManager 读取该插件当前配置
  -> provider.execute(capability, params, context, config)
  -> provider 内部调用第三方库
  -> 返回宿主统一格式
```

### 6.2 provider 是怎么加载的

`plugin.entrypoint` 采用：

```text
模块路径或文件路径:类名
```

例如：

```json
"entrypoint": "./ultimate_provider.py:JavdbProvider"
```

`ProviderManager` 支持两种加载方式：

1. 模块路径导入
2. 相对 manifest 目录的文件路径导入

这意味着插件可以把适配代码完全留在自己目录里，不需要把类注册到宿主里。

### 6.3 当前宿主代码里的典型调用方式

典型调用方式有三类。

#### A. 直接按 capability 调用

例如：

- `ListAppService` 按 `collection.list`、`collection.detail` 拉远程清单
- `VideoAppService` 按 `catalog.detail`、`catalog.by_code` 获取视频详情

这类调用只依赖：

1. 先解析到 manifest
2. 再调用 `gateway.execute_plugin(plugin_id, capability, params)`

#### B. 先拿 client，再按通用方法调用

例如视频运行时支持层 `video_runtime_support.py`：

- `get_video_adapter()`
- `get_playback_proxy_client()`
- `get_preview_request_client()`

这些 client 本质上都是 `ProtocolProviderClient` 或其包装对象，最终还是回到 `provider.execute(...)`。

#### C. 先按协议补展示，再返回前端

例如：

- `annotate_item(...)`
- `annotate_items(...)`

宿主会把 manifest 中声明的 `presentation` 合并到每条记录里，从而把平台展示差异也协议化。

## 7. 当前协议格式说明

每个插件的核心文件是：

```text
ultimate-plugin.json
```

### 7.1 顶层结构

```json
{
  "protocol_version": "1.0",
  "plugin": {},
  "media_types": [],
  "identity": {},
  "presentation": {},
  "capabilities": [],
  "configuration": {},
  "helpers": {},
  "storage": {},
  "collections": {},
  "resource_policy": {}
}
```

### 7.2 顶层字段说明

| 字段 | 是否常用 | 说明 |
| --- | --- | --- |
| `protocol_version` | 必填 | 协议版本 |
| `plugin` | 必填 | 插件基础信息 |
| `media_types` | 必填 | 当前插件作用的内容域，如 `comic`、`video` |
| `identity` | 强烈建议 | 平台标识、ID 前缀、别名 |
| `presentation` | 强烈建议 | 前端展示差异 |
| `capabilities` | 必填 | 插件声明支持的能力列表 |
| `configuration` | 可选 | 配置界面 schema |
| `helpers` | 可选 | 辅助页面、静态资源等 |
| `storage` | 可选 | 数据目录绑定 |
| `collections` | 可选 | 虚拟清单、收藏列表模型 |
| `resource_policy` | 可选 | 资源下载、请求头、刷新策略等 |
| `actions` | 可选 | 补充动作定义，当前较少使用 |

### 7.3 `plugin` 字段

示例：

```json
"plugin": {
  "id": "video.javdb",
  "name": "JAVDB",
  "version": "1.0.0",
  "entrypoint": "./ultimate_provider.py:JavdbProvider",
  "config_key": "javdb"
}
```

说明：

| 字段 | 说明 |
| --- | --- |
| `id` | 插件全局唯一 ID，宿主内部主键 |
| `name` | 展示名 |
| `version` | 插件版本 |
| `entrypoint` | provider 入口 |
| `config_key` | 运行时配置键；如果没有配置项，可以不写 |

注意：

1. `config_key` 为空并不影响插件工作。
2. 没有 `config_key` 的插件通常是“纯能力插件”，例如只做播放代理。

### 7.4 `media_types` 字段

当前已使用：

- `comic`
- `video`

宿主会按内容域筛选插件，例如：

- 漫画搜索只会看 `media_type = comic`
- 视频搜索只会看 `media_type = video`

### 7.5 `identity` 字段

示例：

```json
"identity": {
  "content_type": "video",
  "host_id_prefix": "JAVDB",
  "platform_label": "JAVDB",
  "aliases": ["javdb"]
}
```

主要作用：

1. 声明宿主 ID 前缀，如 `JAVDBV123` 这类前缀解析。
2. 声明平台显示名。
3. 声明查找别名。

建议：

1. `host_id_prefix` 尽量稳定，不要轻易改。
2. `platform_label` 用于前端展示。
3. `aliases` 用于兼容旧调用名与人类输入。

### 7.6 `presentation` 字段

这是“平台展示差异协议化”的关键字段。

当前已使用的字段示例：

```json
"presentation": {
  "media_card": {
    "cover": {
      "aspect_ratio": "16 / 9",
      "mobile_aspect_ratio": "3 / 2",
      "fit": "cover",
      "path_mode": "remote_first"
    },
    "badge": {
      "show_platform_label": true,
      "label": "JAVDB"
    }
  }
}
```

当前宿主已经消费的重点字段：

| 字段 | 作用 |
| --- | --- |
| `media_card.cover.aspect_ratio` | PC 端卡片封面比例 |
| `media_card.cover.mobile_aspect_ratio` | 移动端封面比例 |
| `media_card.cover.fit` | `cover` / `contain` 等显示模式 |
| `media_card.cover.path_mode` | 资源路径模式，如 `local_static` / `remote_first` |
| `media_card.badge.label` | 平台角标文案 |
| `media_card.badge.show_platform_label` | 是否显示平台角标 |

只要是“平台展示差异”，都应该优先考虑继续放到这里。

### 7.7 `capabilities` 字段

示例：

```json
"capabilities": [
  { "key": "catalog.search" },
  { "key": "catalog.detail" },
  { "key": "collection.list" },
  { "key": "collection.detail" }
]
```

每个 capability 至少需要一个 `key`。  
当前已使用的扩展字段还有：

| 字段 | 作用 |
| --- | --- |
| `default_params` | 该能力默认参数 |
| `result_detail_policy` | 告诉宿主搜索结果是否已经足够详细，是否还要继续拉详情 |

当前已经存在的 capability 名称见后文“当前能力清单”。

### 7.8 `configuration` 字段

这是插件自描述配置表单的核心。

示例：

```json
"configuration": {
  "order": 30,
  "label": "JAVDB",
  "sections": [
    {
      "id": "basic",
      "label": "基础配置",
      "fields": [
        { "key": "enabled", "label": "启用", "type": "boolean" },
        { "key": "domain_index", "label": "域名索引", "type": "number" },
        { "key": "cookie_string", "label": "Cookie", "type": "textarea", "secret": true }
      ]
    }
  ],
  "actions": [
    {
      "key": "auth.open_cookie_guide",
      "scope": "configuration",
      "label": "打开 Cookie 获取教程页",
      "kind": "open_url",
      "helper_key": "javdb_cookie_guide"
    }
  ]
}
```

当前表单字段类型已经落地的有：

- `boolean`
- `text`
- `password`
- `textarea`
- `number`

字段常用属性：

| 字段 | 说明 |
| --- | --- |
| `key` | 配置键名 |
| `label` | 显示名称 |
| `type` | 字段类型 |
| `placeholder` | 占位提示 |
| `secret` | 是否敏感信息 |

### 7.9 `helpers` 字段

用于声明插件提供的辅助内容。

当前已落地的 helper 类型：

```json
"helpers": {
  "javdb_cookie_guide": {
    "kind": "static_page",
    "file": "./helpers/cookie_guide/index.html",
    "root_dir": "./helpers/cookie_guide"
  }
}
```

宿主会通过：

```text
/api/v1/config/plugin-helpers/<config_key>/<helper_key>
```

把它暴露出来。

### 7.10 `storage` 字段

用于声明“某个配置字段本质上是数据目录绑定”。

示例：

```json
"storage": {
  "data_dir_bindings": [
    {
      "config_field": "download_dir",
      "relative_dir": "comic/{host_prefix}"
    }
  ]
}
```

作用：

1. 给配置项自动补默认目录。
2. 数据目录迁移时能跟着更新。
3. 减少宿主针对平台目录的硬编码。

### 7.11 `collections` 字段

用于声明插件的收藏夹/清单模型。

当前已使用的能力：

```json
"collections": {
  "list_mode": "virtual_only",
  "virtual_lists": [
    {
      "id": "favorites",
      "name": "我的收藏",
      "description": "平台收藏夹中的所有漫画",
      "capability": "collection.favorites_basic"
    }
  ]
}
```

作用：

1. 宿主可以把没有真实远程 list API 的平台，也包装成通用清单。
2. 清单 UI 不需要知道平台差异。

### 7.12 `resource_policy` 字段

这是资源请求差异协议化的关键字段。

典型用途：

1. 远程图片需要特定 `Referer`
2. 远程资源要带 Cookie
3. 某类资源支持或不支持下载
4. 某类预览链接会过期，需要刷新

示例：

```json
"resource_policy": {
  "assets": {
    "preview_video": {
      "available": true,
      "download_enabled": true,
      "refresh_hint": {
        "mode": "signed_query_expire",
        "match_hosts": ["javdb.com", "jdbstatic.com"],
        "query_param": "t",
        "lead_seconds": 120,
        "refresh_when_missing": true
      },
      "request_profiles": [
        {
          "match_hosts": ["javdb.com", "jdbstatic.com"],
          "referer": "https://javdb.com/",
          "cookie_config_key": "javdb",
          "cookie_field_path": "cookies"
        }
      ]
    }
  }
}
```

当前宿主已经消费的重点字段：

| 字段 | 作用 |
| --- | --- |
| `assets.<kind>.available` | 是否存在该类资源 |
| `assets.<kind>.download_enabled` | 是否允许下载 |
| `assets.<kind>.url_base` | 相对资源路径拼接基础地址 |
| `assets.<kind>.request_profiles` | 资源请求策略 |
| `assets.<kind>.refresh_hint` | 预览链接刷新策略 |

`request_profiles` 当前支持的关键字段：

| 字段 | 作用 |
| --- | --- |
| `match_hosts` | 按 host 匹配资源 |
| `path_prefixes` | 按 URL 前缀匹配资源 |
| `referer` | 固定 Referer |
| `referer_template` | 可按内容 ID 模板化 Referer |
| `origin` | 固定 Origin |
| `cookie_config_key` | 从哪个插件配置里取 Cookie |
| `cookie_field_path` | Cookie 字段路径 |
| `headers` | 额外请求头 |
| `frontend_proxy` | 前端代理提示 |

## 8. 当前已落地的 capability 清单

当前宿主已经在使用的 capability 主要有这些：

| capability | 说明 |
| --- | --- |
| `catalog.search` | 搜索内容 |
| `catalog.detail` | 获取详情 |
| `catalog.by_code` | 按番号/编码查视频 |
| `person.search` | 搜索演员/人物 |
| `person.works` | 获取演员作品 |
| `collection.favorites` | 获取完整收藏夹 |
| `collection.favorites_basic` | 获取轻量收藏夹 |
| `collection.list` | 获取远程清单列表 |
| `collection.detail` | 获取远程清单详情 |
| `taxonomy.tags` | 获取标签体系 |
| `taxonomy.tag_search` | 按标签组合搜索 |
| `asset.bundle.fetch` | 下载内容包，如整本漫画 |
| `asset.cover.fetch` | 下载封面 |
| `asset.preview.resolve` | 解析预览图 URL |
| `storage.comic_dir.resolve` | 解析漫画下载目录 |
| `playback.sources.build` | 构建在线播放源 |
| `playback.proxy.stream` | 代理流式资源 |
| `playback.proxy.url` | 代理普通 URL 请求 |
| `transport.http.request` | 用插件自己的请求栈发 HTTP 请求 |
| `health.query.status` | 返回当前平台查询凭据是否就绪 |

### 8.1 宿主对结果格式的最小期待

下面是“当前宿主真正依赖的最小约定”。  
插件可以返回更多字段，但最好至少覆盖这些关键字段。

| capability | 最小返回约定 |
| --- | --- |
| `catalog.search` 漫画 | 返回字典，包含 `albums` 列表；每项至少有 `album_id`、`title`，通常还会有 `author`、`cover_url` |
| `catalog.search` 视频 | 返回字典，包含 `videos` 列表；部分链路也兼容 `works` |
| `catalog.detail` 漫画 | 返回漫画详情或包含单条详情的字典，宿主会继续做统一转换 |
| `catalog.detail` 视频 | 返回单条视频详情字典 |
| `collection.list` | 返回字典，包含 `lists` 列表 |
| `collection.detail` | 返回字典，漫画侧一般包含 `albums`，视频侧一般包含 `works` 或 `videos` |
| `asset.bundle.fetch` | 返回 `{ "detail": {...}, "success": true/false }` |
| `asset.cover.fetch` | 返回 `{ "detail": {...}, "success": true/false }` |
| `asset.preview.resolve` | 返回 `List[str]` |
| `storage.comic_dir.resolve` | 返回字符串目录路径 |
| `health.query.status` | 返回 `{ configured, message, missing_fields }` 风格结构 |

### 8.2 为什么 capability 必须先声明

宿主很多地方不是“先写死平台，再试试看”，而是：

1. 先筛选“声明了某个 capability 的插件”
2. 再把它纳入业务流程

例如：

- 视频搜索只枚举声明了 `catalog.search` 的视频插件
- 播放代理客户端会找声明了 `playback.proxy.stream` 的插件
- 预览请求客户端会找声明了 `transport.http.request` 的插件

所以如果一个插件没有在 manifest 中声明能力，即使 provider 里实现了，宿主也不会把它纳入对应流程。

## 10. 新增一个平台，但只复用已有功能，应该怎么做

目标是：

1. 不改宿主业务代码。
2. 不改前端平台分支代码。
3. 差异尽量下放到插件目录。

### 10.1 标准步骤

1. 在 `comic_backend/third_party` 下创建插件目录。
2. 添加 `ultimate-plugin.json`。
3. 编写 provider 类，继承 `ProtocolProvider`。
4. 在 provider 中实现 manifest 已声明的 capability。
5. 如果平台有配置项，声明 `plugin.config_key` 和 `configuration`。
6. 如果平台有展示差异，写入 `presentation`。
7. 如果平台有资源请求差异，写入 `resource_policy`。
8. 如果平台有收藏夹映射差异，写入 `collections`。
9. 如果平台需要帮助页或配置交互，写入 `helpers + configuration.actions`。
10. 补测试。

### 10.2 最小 manifest 示例


### 10.3 最小 provider 示例

```python
from typing import Any, Dict

from protocol.base import ProtocolProvider


class DemoVideoProvider(ProtocolProvider):
    def normalize_config(self, payload: Dict[str, Any]) -> Dict[str, Any]:
        normalized = dict(payload or {})
        normalized.setdefault("enabled", True)
        return normalized

    def get_query_status(self, config: Dict[str, Any]) -> Dict[str, Any]:
        token = str((config or {}).get("token") or "").strip()
        if token:
            return {"configured": True, "message": "", "missing_fields": []}
        return {"configured": False, "message": "缺少 token", "missing_fields": ["token"]}

    def execute(self, capability: str, params: Dict[str, Any], context: Dict[str, Any], config: Dict[str, Any]):
        if capability == "catalog.search":
            return {"page": 1, "has_next": False, "videos": []}
        if capability == "catalog.detail":
            return {}
        if capability == "health.query.status":
            return self.get_query_status(config)
        raise ValueError(f"unsupported capability: {capability}")
```

### 10.4 新增平台时最重要的原则

1. 只声明你真的支持的 capability。
2. 不要为了“凑统一”去声明一个实际上没有意义的能力。
3. 平台差异优先写进协议字段，而不是要求宿主再加分支。
4. 如果资源请求需要特殊头、Referer、Cookie，请优先写入 `resource_policy`。
5. 如果封面比例、角标、路径模式不同，请优先写入 `presentation`。

## 11. 为一个已有平台新增“已有功能”，应该怎么做

这里指的是：

宿主已经存在某个能力，例如：

- `taxonomy.tags`
- `collection.list`
- `asset.cover.fetch`

但某个已有插件原来没做，现在想补上。

### 11.1 操作方式

1. 在该插件的 manifest 中新增 capability 声明。
2. 在 provider 的 `execute()` 中实现这条 capability。
3. 如果需要额外配置、资源策略、展示字段，也补进 manifest。
4. 补测试。

### 11.2 什么时候不需要改宿主

只要宿主已经有对应 capability 的通用调用链，就不需要改宿主。

例如：

1. 给某个视频平台新增 `catalog.by_code`
2. 给某个漫画平台新增 `collection.list`
3. 给某个平台新增 `asset.cover.fetch`

这类扩展通常只需要改插件。

### 11.3 什么时候虽然是“旧功能”，仍可能要补宿主

如果宿主当前还没有把某个通用 capability 真正接进某条业务链，那就需要做一次“宿主的通用接线”，但仍然不应该写平台特判。

正确做法是：

1. 在宿主新增“通用 capability 调用链”
2. 所有平台都按同一个 capability 协议接入

错误做法是：

1. 新增 `/api/v1/video/third-party/some-platform/special-route`
2. 在业务代码里写 `if platform == xxx`

## 12. 如果要扩展一个全新的功能域，应该怎么做

例如以后要支持：

- 字幕抓取
- 弹幕导入
- 章节评论同步
- 种子管理

这类能力当前宿主还没有。

### 12.1 正确扩展方式

应该一次性做成“新的通用 capability 协议”，而不是为某个平台写死一套逻辑。

步骤建议：

1. 先给能力命名，定义统一 capability key。
2. 定义统一入参与返回结构。
3. 在宿主加一个通用调用入口。
4. 在需要的平台插件中声明并实现它。
5. 把展示差异、配置差异、资源差异继续放在协议字段里。
6. 补全协议测试与业务测试。

### 12.2 一个简单判断标准

如果你准备在宿主写：

```python
if platform == "javdb":
    ...
elif platform == "javbus":
    ...
```

那通常说明这件事还没有被协议化，应该先退一步重新设计 capability 或协议字段。

## 13. 推荐的开发与测试门禁

为了保证“插件化重构后功能不变”，新增插件或扩展协议时建议至少覆盖两层测试：

### 13.1 协议层测试

建议放在：

```text
comic_backend/tests/
```

重点验证：

1. manifest 能被扫描和解析
2. provider 能被动态加载
3. runtime config 会正确合入默认值
4. 兼容查找名与 capability 过滤正常

### 13.2 业务集成测试

建议放在：

```text
tests/features/third_party_integration/integration/
```

重点验证：

1. 搜索、详情、收藏、清单等宿主功能不变
2. 资源下载、代理、预览视频刷新等链路不变
3. 展示信息和资源策略被宿主正确消费

## 14. 当前架构下的最佳实践

### 14.1 应该放进插件协议里的东西

1. 平台名称、别名、前缀
2. 能力声明
3. 配置字段
4. 展示差异
5. 资源请求差异
6. 虚拟列表/收藏夹模型
7. 配置辅助页

### 14.2 不应该继续放进宿主里的东西

1. 平台专属封面比例判断
2. 平台专属 Cookie 教程路由
3. 平台专属资源头部拼接
4. 平台专属默认目录规则
5. 平台专属功能入口判断

### 14.3 provider 内部允许做什么

provider 内部允许并且应该保留对第三方库本身的耦合，例如：

1. 调库方式
2. 数据结构转换
3. 登录态处理
4. 反爬请求栈
5. 平台专有解析逻辑

这些耦合应该被限制在插件目录内部，不应该继续泄漏到宿主。

## 15. 一句话总结

当前第三方体系已经不是“宿主内置几个平台适配器”，而是：

```text
宿主负责扫描 manifest、加载 provider、管理配置、按 capability 调度；
插件负责声明自己是谁、支持什么、怎么配、怎么展示、怎么请求资源、以及如何真正调用第三方库。
```

后续新增平台时，优先新增插件；后续新增能力时，优先新增通用 capability 协议。  
这样才能长期保持“宿主面向协议，平台耦合留在插件内部”。
