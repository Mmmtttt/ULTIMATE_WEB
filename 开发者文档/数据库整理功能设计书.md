# 数据库整理功能设计书（漫画/视频双模式可扩展）

## 1. 背景与目标

当前项目“整理数据库”只有单一动作（修复封面+页数回写），且入口在系统设置页，不满足以下诉求：

1. 入口迁移到“我的”页面。
2. 点击后展示多个子功能，而不是直接执行。
3. 漫画模式与视频模式的子功能可不同。
4. 现有“整理数据库”动作重命名为“修复封面”。
5. 新增漫画模式子功能：
   - 本地库/预览库按“清洗标题 + 章节规则”去重，重复项移入回收站。
   - LOCAL 漫画补全信息（JM 优先，JM 不匹配再 PK），并做幂等跳过。
6. 为视频模式后续扩展预留统一架构。

---

## 2. 设计原则

1. **统一入口**：前端只维护一个“数据库整理”入口。
2. **模式驱动**：后端返回当前 mode 的可执行选项，前端动态渲染。
3. **动作解耦**：`mode + action` 路由到具体实现，便于后续扩展视频动作。
4. **安全优先**：去重不做永久删除，只置回收站标记。
5. **幂等可重复执行**：LOCAL 补全成功后打标，后续自动跳过。
6. **兼容旧接口**：保留 `/api/v1/comic/organize`，语义映射为 `comic/repair_cover`。

---

## 3. 前端方案

### 3.1 入口位置

- 新入口：`Mine.vue`（“我的”页面）新增 `数据库整理` cell。
- 点击后弹出 action 列表（动作来自后端 options 接口）。

### 3.2 模式差异

- 漫画模式：显示 3 个动作
  - `repair_cover`：修复封面（原整理数据库）
  - `deduplicate_by_title`：按标题去重并移入回收站
  - `enrich_local_metadata`：LOCAL 漫画补全信息
- 视频模式：返回占位动作（当前不可执行），前端显示“建设中/待实现”。

### 3.3 交互流程

1. 打开面板前调用 `GET /api/v1/organize/options?mode={comic|video}`。
2. 用户选择动作后弹确认框。
3. 确认后调用 `POST /api/v1/organize/run`。
4. 后端返回 `summary`，前端 toast 展示。

### 3.4 旧入口处理

- `SystemConfig.vue` 内原“整理数据库”按钮移除，避免双入口歧义。

---

## 4. 后端方案

### 4.1 新增应用服务

新增 `application/database_organize_service.py`：

- `get_options(mode)`：返回某模式可用动作列表。
- `run(mode, action)`：统一分发执行。

动作分发示意：

- comic
  - `repair_cover` -> `ComicAppService.organize_database_v2()`
  - `deduplicate_by_title` -> `ComicAppService.organize_deduplicate_by_title()`
  - `enrich_local_metadata` -> `ComicAppService.organize_enrich_local_metadata()`
- video
  - 当前返回未实现提示，保留扩展位。

### 4.2 新增 API

新增 `api/v1/organize.py`：

1. `GET /api/v1/organize/options?mode=comic|video`
2. `POST /api/v1/organize/run`
   - body: `{ "mode": "comic", "action": "deduplicate_by_title" }`

### 4.3 兼容旧 API

`POST /api/v1/comic/organize` 保留，内部改为执行 `comic + repair_cover`，保证旧前端/外部调用不崩。

## 5. 漫画动作详细设计

### 5.1 动作一：修复封面（原整理数据库）

直接复用 `organize_database_v2`。

### 5.2 动作二：本地库/预览库去重

#### 5.2.1 作用范围

- 本地库：`comics_database.json` 的 `comics`
- 预览库：`recommendations_database.json` 的 `recommendations`
- 仅处理 `is_deleted != true` 的记录。

#### 5.2.2 标题清洗规则

对标题执行：

1. 删除各种括号及其中内容：`()` `（ ）` `[]` `【 】` `{}` `<>` 等。
2. 删除所有空白字符（空格/制表/换行）。
3. 小写化（用于英文对齐）。

#### 5.2.3 章节信息规则

识别常见章节标识（如 `第X章/话/卷`、`上卷/下卷`、`vol/ch/chapter` 等）。

去重判定键：

- `clean_title + chapter_signature`

判重原则：

1. 清洗后标题相同且章节签名相同 -> 判定重复。
2. 清洗后标题相同但章节签名不同 -> **不重复**。
3. 一条有章节、一条无章节 -> **不重复**（保守策略，避免误删）。

#### 5.2.4 冲突处理

- 每个重复组保留 1 条（按当前顺序首条保留）。
- 其余条目置 `is_deleted=true`（移入回收站）。
- 不做物理删除、不删文件。

#### 5.2.5 返回结果

返回 home/recommendation 各自统计：

- 扫描数、重复组数、移入回收站数、保留数、跳过数。
- 汇总 `summary` 供前端弹窗。

### 5.3 动作三：LOCAL 漫画补全信息

#### 5.3.1 作用范围

- 仅本地库 `comics`。
- 仅 `LOCAL*` 漫画 ID。
- 仅 `is_deleted != true`。

#### 5.3.2 幂等标志位

新增 comic 字段：`local_metadata_enriched: bool`。

规则：

- 该字段为 `true` 的记录直接跳过。
- 本次成功匹配并写回后置为 `true`。

#### 5.3.3 匹配流程

对每条候选 LOCAL 记录：

1. 计算本地清洗标题 + 章节签名。
2. JM 搜索（关键词使用清洗标题），取**第一个结果**。
3. 若 JM 第一个结果清洗标题+章节签名匹配 -> 认定命中。
4. 若 JM 不命中，再按同样规则尝试 PK。
5. 两者都不命中 -> 跳过。

#### 5.3.4 明细补全

命中后调用详情接口，写回：

- `author`：远端 author 非空则覆盖本地 author。
- `tag_ids`：按“标签名称 -> 标签ID”映射追加（去重）
  - 标签库无该名称：创建新 comic 标签（`content_type=comic`）
  - 已有：直接复用 ID
- 置 `local_metadata_enriched=true`。

#### 5.3.5 失败与降级

- 单条失败不影响整体任务，累计 `failed_count`。
- 若 third-party 运行时关闭，返回可读错误信息或全跳过统计。

## 6. 数据模型变更

`domain/comic/entity.py` 增加字段：

- `local_metadata_enriched: bool = False`

并同步 `from_dict/to_dict`，避免后续 repository 保存时丢字段。

## 7. 测试设计（集成）

### 7.1 选项接口

- `mode=comic` 返回 3 个动作。
- `mode=video` 返回视频占位动作（不可执行）。

### 7.2 去重动作

构造 home/recommendation 重复数据：

- 验证同标题同章节会被去重。
- 验证同标题不同章节不会误删。
- 验证重复项仅 `is_deleted=true`，未永久删除。

### 7.3 LOCAL 补全动作

在 third_party 集成夹具中 mock 平台搜索与详情：

- 验证 JM 优先，JM 不匹配时回退 PK。
- 验证 author/tag_ids 写回。
- 验证标签复用与新建逻辑。
- 验证 `local_metadata_enriched=true` 后下次跳过。

## 8. 未来扩展（视频模式）

后续新增视频动作时，仅需：

1. 在 `DatabaseOrganizeService` 注册 `video` 动作。
2. 实现 `VideoAppService` 对应方法。
3. options 接口返回新动作元数据。

前端无需改交互结构。
