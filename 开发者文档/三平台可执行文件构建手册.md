# 三平台可执行文件构建手册

更新时间：2026-03-19

本文档目标：
- 新开发者从 `git clone` 开始，完成环境准备、开发和出包。
- 使用统一代码库开发，默认不感知 Windows/Linux/Android 平台差异。
- 通过统一流水线产出三类包：`Windows EXE`、`Linux 可执行文件`、`Android APK`。

## 1. 总体结论

1. Windows 主机可直接构建：`Windows EXE + Android APK`。
2. Linux 可执行文件必须在 Linux 主机或 Linux CI Runner 构建。
3. 想“一次触发拿到三平台产物”，推荐使用 CI 工作流统一编排。
4. Android 当前已支持：
- APK 启动即拉起内置 Python 后端。
- 前端默认连接 `127.0.0.1:5000`。
- 后端数据落在 Android 可写目录（由运行时环境变量注入）。

## 2. 从零开始（克隆仓库）

```powershell
git clone https://github.com/Mmmtttt/ULTIMATE_WEB.git
cd ULTIMATE_WEB
git submodule update --init --recursive
```

## 3. Windows（Conda base）环境安装

```powershell
conda activate base
powershell -ExecutionPolicy Bypass -File .\scripts\install_build_env_windows_base.ps1
```

安装完成后，关闭并重新打开终端，再执行：

```powershell
conda activate base
python --version
node --version
npm --version
npx --version
java --version
sdkmanager.bat --version
adb --version
python -c "import importlib.util; print('PyInstaller=', bool(importlib.util.find_spec('PyInstaller')), 'flask_cors=', bool(importlib.util.find_spec('flask_cors')))"
```

## 4. 日常开发（统一代码库）

```powershell
git checkout -b feat/<分支名>
# 修改前后端代码
git add .
git commit -m "feat: <说明>"
```

说明：
- 业务代码只维护一份，不需要给三个平台分别写逻辑。
- 平台差异在构建/打包阶段由统一脚本处理。

## 5. Windows 本地出包（Windows + Android）

### 5.1 一条命令完成构建+打包

```powershell
conda activate base
python .\scripts\release_unified.py --targets "windows,android" --execute --build-output output\multi_target --package-output output\packages
```

### 5.2 产物位置

- Windows 主产物：`output/packages/windows/dist/ultimate_backend_windows.exe`
- Windows 发布包副本：`output/packages/windows/release_bundle/bin/ultimate_backend_windows.exe`
- Android 产物：`output/packages/android/apk/app-debug.apk`
- 打包汇总：`output/packages/packaging_summary.json`

## 6. Linux 产物构建（Linux 主机或 Linux CI）

```bash
git clone https://github.com/Mmmtttt/ULTIMATE_WEB.git
cd ULTIMATE_WEB
git submodule update --init --recursive

python3 -m pip install --upgrade pip
python3 -m pip install -r comic_backend/requirements.txt pyinstaller
npm ci --prefix comic_frontend

python3 scripts/release_unified.py --targets linux --execute --build-output output/multi_target --package-output output/packages
```

产物位置：
- `output/packages/linux/dist/ultimate_backend_linux`
- `output/packages/linux/release_bundle/bin/ultimate_backend_linux`

## 7. 一次触发三平台（推荐 CI）

### 7.1 推送代码并触发工作流

```bash
git checkout -b feat/<分支名>
# 修改代码
git add .
git commit -m "feat: <说明>"
git push origin feat/<分支名>

gh auth login
gh workflow run release-three-platforms.yml --ref feat/<分支名>
RUN_ID=$(gh run list --workflow release-three-platforms.yml --branch feat/<分支名> --limit 1 --json databaseId --jq '.[0].databaseId')
gh run watch "$RUN_ID"
```

### 7.2 下载三平台产物

```bash
mkdir -p output/from_ci
gh run download "$RUN_ID" -n ultimate-release-bundle -D output/from_ci
```

常见路径：
- `output/from_ci/ultimate-windows/dist/ultimate_backend_windows.exe`
- `output/from_ci/ultimate-linux/dist/ultimate_backend_linux`
- `output/from_ci/ultimate-android/apk/app-debug.apk`
- `output/from_ci/RELEASE_MANIFEST_SHA256.txt`

## 9. 常见问题与处理

### 9.1 Android 构建报错 `SDK location not found`

- 打包脚本会自动写入 `android/local.properties`。
- 若仍报错，请确认：

```powershell
conda activate base
sdkmanager.bat --version
```

### 9.2 `adb` / `sdkmanager` 命令找不到

先确认已重新打开终端并执行：

```powershell
conda activate base
```

再检查 `D:\environment\conda\etc\conda\activate.d\android_sdk_env.ps1` 是否为合法多行 PowerShell 脚本。

### 9.3 Android 打包锁文件问题（`.gradle/.../checksums.lock`）

- 当前构建会使用独立 workspace，已尽量避免复用锁。
- 若仍偶发，先结束残留 `java.exe` 进程再重试。

## 10. 后端增量定向同步（局域网配对）

本节是当前新增能力：支持“定向 + 增量 + 并集合并”。

### 10.1 同步语义

- A 数据：`1,2,3`
- B 数据：`2,3,4`
- 执行 A -> B 后，B 结果：`1,2,3,4`

规则：
- 默认按实体 `id` 做并集合并。
- 已存在记录不做覆盖更新（仅补空字段，`tag_ids/list_ids/actors` 做并集）。
- 方向可控：`push`（本机推送到对端）或 `pull`（从对端拉取到本机）。

### 10.2 局域网配对流程

#### 在 B 上创建配对码

```powershell
$body = @{ ttl_minutes = 10 } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "http://<B-IP>:5000/api/v1/sync/pairing/invite" -ContentType "application/json" -Body $body
```

返回 `pairing_code`。

#### 在 A 上发起连接（输入 B 地址 + 配对码）

```powershell
$body = @{
  remote_base_url = "http://<B-IP>:5000"
  pairing_code = "<B返回的pairing_code>"
} | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "http://<A-IP>:5000/api/v1/sync/pairing/connect" -ContentType "application/json" -Body $body
```

#### 在 A 上查看已配对 peer

```powershell
Invoke-RestMethod -Method Get -Uri "http://<A-IP>:5000/api/v1/sync/peers"
```

### 10.3 执行定向增量同步

#### A -> B（push）

```powershell
$body = @{ peer_id = "<A上的peer_id>" } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "http://<A-IP>:5000/api/v1/sync/directional/push" -ContentType "application/json" -Body $body
```

#### B -> A（在 A 上 pull）

```powershell
$body = @{ peer_id = "<A上的peer_id>" } | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "http://<A-IP>:5000/api/v1/sync/directional/pull" -ContentType "application/json" -Body $body
```

### 10.4 相关后端接口一览

1. `POST /api/v1/sync/pairing/invite`
- 创建局域网配对码。

2. `POST /api/v1/sync/pairing/claim`
- 对端使用配对码完成 claim（由 connect 内部调用）。

3. `POST /api/v1/sync/pairing/connect`
- 建立本机到对端的配对关系并保存 token。

4. `GET /api/v1/sync/peers`
- 查询已配对 peer 列表。

5. `DELETE /api/v1/sync/peers/<peer_id>`
- 删除某个 peer。

6. `POST /api/v1/sync/directional/push`
- 本机数据增量推送到 peer。

7. `POST /api/v1/sync/directional/pull`
- 从 peer 增量拉取到本机。

8. `GET /api/v1/sync/directional/inventory`（token 鉴权）
- 获取设备数据清单。

9. `POST /api/v1/sync/directional/delta`（token 鉴权）
- 基于对端已知清单生成增量。

10. `POST /api/v1/sync/directional/apply`（token 鉴权）
- 应用增量到本地数据。

11. `GET /api/v1/sync/directional/assets/inventory`（token 鉴权）
- 获取图片/缓存等资产文件清单（相对路径 + 签名）。

12. `POST /api/v1/sync/directional/assets/apply`（token 鉴权）
- 接收并应用资产增量包（zip）。

13. `POST /api/v1/sync/directional/assets/delta/download`（token 鉴权）
- 根据对端已知文件清单，生成并返回资产增量 zip（无增量时返回 204）。

### 10.5 当前增量算法说明

1. 结构化数据（JSON 库）
- 以实体 `id` 为主键做差集：只传对端缺失的记录。
- 合并策略是“并集优先”：不覆盖对端已有内容，仅补空字段；`tag_ids/list_ids/actors` 做并集。

2. 资产文件（图片/缓存）
- 构建资产索引：`相对路径 -> size:mtime`。
- 对比对端索引后，只打包签名不一致或缺失的文件。
- 通过 zip 传输并在目标端安全解压到 `DATA_DIR` 下的允许目录（`comic/`、`video/`、`static/`、`cache/`、`recommendation_cache/`）。
